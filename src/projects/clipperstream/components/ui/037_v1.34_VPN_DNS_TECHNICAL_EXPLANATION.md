## The Answer: DNS Caching & Network State

### What Happened (Timeline)

#### Scenario 1: Online Recording (Success) ✅

```
Time 0:00 - User loads app (VPN ON)
Time 0:01 - User records audio
Time 0:02 - App calls Deepgram API
          ↓
          DNS lookup for api.deepgram.com
          ↓
          VPN DNS resolver SUCCEEDS (lucky timing)
          ↓
          Result: 35.186.224.25 (Deepgram IP)
          ↓
          DNS result cached by:
          - Browser/Node.js (for ~60-300 seconds)
          - Operating System (for ~60-300 seconds)
          - VPN client (for ~60-300 seconds)
          ↓
Time 0:03 - API call succeeds ✅
Time 0:04 - Transcription completes ✅
```

**Why it worked**:
- VPN DNS resolved successfully (might have been working at that moment)
- Result got cached
- Subsequent immediate calls reuse the cached IP
- No new DNS lookup needed

---

#### Scenario 2: Offline → Online Retry (Failed) ❌

```
Time 0:00 - User loads app (VPN ON)
Time 0:01 - User records online (DNS cache populated) ✅
Time 5:00 - User goes OFFLINE
          ↓
          Network disconnects
          ↓
          All TCP connections close
          ↓
          DNS cache might be invalidated

Time 5:05 - User records audio (stored in IndexedDB)

Time 10:00 - User goes back ONLINE
          ↓
          Network reconnects
          ↓
          VPN reconnects/refreshes
          ↓
          DNS cache expired (or cleared on network change)

Time 10:01 - Auto-retry triggered
          ↓
          FRESH DNS lookup needed (cache expired)
          ↓
          VPN DNS resolver FAILS (blocked or timeout)
          ↓
          Error: ENOTFOUND api.deepgram.com ❌
```

**Why it failed**:
- Time gap (5-10 minutes) caused DNS cache to expire
- Network disconnect/reconnect cleared cached IPs
- VPN DNS server blocked or failed to resolve
- No cached IP to fall back to

## Technical Deep Dive

### 1. DNS Caching Layers

**Multiple caches between your app and the actual DNS server**:

```
Your App (Node.js)
    ↓ (checks cache: ~60s TTL)
Operating System DNS Cache
    ↓ (checks cache: ~60-300s TTL)
VPN Client
    ↓ (checks cache: varies)
VPN DNS Server
    ↓ (might block/filter)
Actual DNS Server (8.8.8.8, 1.1.1.1, etc.)
```

**On first online recording**:
- All caches empty
- Full DNS resolution happens
- VPN allows it (maybe it's in a "good" state)
- Result cached at all levels
- Subsequent calls use cache

**On offline → online retry**:
- Network disconnect cleared caches
- OR cache TTL expired
- Fresh DNS lookup needed
- VPN now blocks/fails (maybe server changed, or VPN rules kicked in)

---

### 2. VPN DNS Behavior Patterns

**Why VPN DNS might be intermittent**:

#### A. DNS Server Rotation
```
First request  → VPN DNS Server A (works) ✅
Second request → VPN DNS Server B (blocks Deepgram) ❌
```

Many VPNs rotate DNS servers for load balancing. Different servers might have different filtering rules.

#### B. Rate Limiting
```
First request  → Under rate limit (works) ✅
Second request → Rate limit hit (fails) ❌
```

Some VPN DNS servers rate-limit lookups to prevent abuse.

#### C. Time-Based Filtering
```
Initial connection → Allowlist mode (permissive) ✅
After 5 minutes   → Blocklist active (strict) ❌
```

Some VPNs have "grace period" after connection before full filtering kicks in.

#### D. Domain Categorization
```
api.deepgram.com might be categorized as:
- "AI/ML service"
- "API endpoint"
- "Third-party tracking" (incorrectly)
```

VPN might allow first lookup but block subsequent ones after categorizing it.

---

### 3. Network State Changes

**What happens when you go offline → online**:

```
OFFLINE EVENT:
├─ All TCP connections terminated
├─ DNS cache invalidated (OS clears on network change)
├─ VPN connection paused/suspended
└─ Socket pools cleared

ONLINE EVENT:
├─ Network interface reinitialized
├─ VPN reconnects (might get different server)
├─ New DNS resolver assigned
├─ Fresh DNS lookups required
└─ No cached IPs available
```

**Result**: Everything starts fresh, no cached DNS entries to fall back on.

---

### 4. Why Immediate Online Calls Work

**Connection pooling and DNS caching**:

```javascript
// First API call (online recording)
const response1 = await fetch('https://api.deepgram.com/v1/listen', ...);
// DNS lookup: api.deepgram.com → 35.186.224.25
// Result cached
// Connection kept alive (HTTP keep-alive)

// If you immediately called again (within ~60s):
const response2 = await fetch('https://api.deepgram.com/v1/listen', ...);
// NO DNS lookup! Uses cached IP: 35.186.224.25
// Reuses existing TCP connection
```

**But after offline → online**:

```javascript
// Network disconnected → all connections closed
// DNS cache invalidated

// Offline recording retry (minutes later)
const response = await fetch('https://api.deepgram.com/v1/listen', ...);
// FRESH DNS lookup needed
// VPN DNS fails → ENOTFOUND
```

## Why Different VPNs Behave Differently

### Free VPNs (Most Aggressive)
- Heavy DNS filtering
- Block "tracking" domains (might include APIs)
- Inconsistent DNS servers
- **Impact**: High failure rate

### Premium VPNs (Less Aggressive)
- Some DNS filtering, but configurable
- More reliable DNS servers
- Better infrastructure
- **Impact**: Medium failure rate

### Corporate VPNs (Policy-Based)
- Strict security policies
- Might block external APIs
- Whitelist/blacklist based
- **Impact**: High failure rate (if not whitelisted)

### "Privacy-Focused" VPNs
- DNS-level ad/tracker blocking
- Might categorize `api.deepgram.com` as tracker
- **Impact**: Medium-high failure rate

      return await fetch(url, options);

      if (err.message.includes('ENOTFOUND')) {
        // Wait and retry: 1s, 2s, 4s

## Technical Answer to "Why?"

**In one sentence**:

> The first online recording succeeded because the DNS result was cached from a successful lookup, but after going offline and back online, the network state change invalidated the DNS cache, requiring a fresh DNS lookup that the VPN's DNS server blocked or failed to resolve.

**The key insight**:

DNS caching creates a "grace period" where subsequent calls work, but once that cache expires (due to time or network changes), fresh DNS lookups are required, which is when VPN DNS issues manifest.
