## Root Cause: The Real Issue

### The Actual Problem Flow

1. **Recording stops** → `mediaRecorder.onstop` fires
2. **Blob created** with correct MIME type: `blob.type = 'audio/webm;codecs=opus'`
3. **Blob saved to IndexedDB** via `storeAudio(blob)`
4. **Blob's `type` property is lost** during IndexedDB serialization
5. **Later retrieval** → Blob has `type: ''` (empty string)
6. **Transcription API rejects** → 500 error

### Why IndexedDB Loses the Type

IndexedDB serializes Blobs as-is, but some browsers (especially Safari) don't preserve the `type` property during the round-trip. This is a **known browser bug**.

### The Correct Solution

**Option A: Store MIME type separately during save**

In `useClipRecording.ts`, when saving:
```typescript

  savedAudioId = await storeAudio(blob);
  
  // Also store the MIME type
  setRecordingMimeType(mimeType);  // ← New state
  
  // ... rest of code ...
};
```

Then pass it through to `handleOfflineRecording`:

**Option B: Fix `storeAudio()` to preserve MIME type metadata**

Modify `audioStorage.ts` to store MIME type alongside blob:
