## Flow Comparison: Online vs Offline

### Online Flow (Works ✅)
```
1. MediaRecorder creates blob
   → blob.type = "audio/webm;codecs=opus"

2. storeAudio() saves to IndexedDB
   → { blob, mimeType: "audio/webm;codecs=opus" }

3. transcribeRecording() called immediately
   → Uses SAME blob instance from step 1
   → blob.type still = "audio/webm;codecs=opus"

4. FormData.append('audio', blob, 'recording.webm')
   → Browser reads blob.type for Content-Type header
   → Content-Type: audio/webm;codecs=opus

5. Server receives multipart form
   → formidable parses with correct MIME type
   → Deepgram transcribes successfully ✅
```

### Offline Flow (Fails ❌)
```
1. MediaRecorder creates blob
   → blob.type = "audio/webm;codecs=opus"

2. storeAudio() saves to IndexedDB
   → { blob, mimeType: "audio/webm;codecs=opus" }

3. [USER GOES OFFLINE, THEN BACK ONLINE]

4. getAudio() retrieves from IndexedDB
   → result.blob.type = "" (BROWSER BUG)
   → Creates: new Blob([result.blob], { type: "audio/webm" })
   → correctedBlob.type = "audio/webm"

5. transcribeRecording() called with retrieved blob
   → blob.type = "audio/webm" (NOT "audio/webm;codecs=opus")

6. FormData.append('audio', blob, 'recording.webm')
   → Content-Type: audio/webm (missing codecs)

7. Server receives multipart form
   → formidable parses with MIME type: "audio/webm"
   → **500 ERROR** ❌
```

## Root Cause Hypothesis

### Hypothesis 1: MIME Type Mismatch (Likely)
**Issue**: Retrieved blob has `audio/webm` instead of `audio/webm;codecs=opus`

**Evidence**:
- [audioStorage.ts:69](../../../services/audioStorage.ts#L69) stores fallback: `blob.type || 'audio/webm'`
- [audioStorage.ts:107](../../../services/audioStorage.ts#L107) retrieves fallback: `result.mimeType || 'audio/webm'`
- **Missing codec information** when recreating blob

**Why this causes 500**:
- Deepgram expects specific codec info for proper decoding
- Generic `audio/webm` without codec might cause transcription failure
- Server logs would show: "Deepgram API error: [specific error]"

**Fix**: Store full MIME type including codecs
```typescript
// CURRENT (WRONG):
mimeType: blob.type || 'audio/webm'

// SHOULD BE:
mimeType: blob.type || 'audio/webm;codecs=opus'
```

### Hypothesis 2: Blob Reconstruction Fails (Possible)
**Issue**: `new Blob([result.blob], { type: mimeType })` doesn't properly reconstruct blob

**Evidence**:
- IndexedDB might deserialize Blob differently across browsers
- Creating a Blob from another Blob might not preserve binary data correctly

**Why this causes 500**:
- Blob content is corrupted during reconstruction
- Server receives invalid/empty audio data
- Server logs would show: "No audio recorded" or "audioData.length === 0"

**Fix**: Use ArrayBuffer instead
```typescript
// Store as ArrayBuffer
const arrayBuffer = await blob.arrayBuffer();
store.add({ id: audioId, arrayBuffer, mimeType: blob.type });

// Retrieve and recreate as Blob
const blob = new Blob([result.arrayBuffer], { type: result.mimeType });
```

### Hypothesis 3: Server-Side Issue (Unknown)
**Issue**: Server rejects audio for a reason unrelated to MIME type

**Possible causes**:
- Deepgram API key issue (but online transcription works)
- File size validation (but stored blob size looks correct)
- Temporary file system error
- Deepgram API rate limiting
- Network timeout during retrieval

**Evidence needed**: Server-side console logs
