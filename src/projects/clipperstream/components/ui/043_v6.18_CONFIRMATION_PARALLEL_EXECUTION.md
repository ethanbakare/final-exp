## Why This Works (Technical Details)

### 1. JavaScript is Single-Threaded, but Asynchronous

```typescript
// Background process
async function processAllPendingClips() {
  const result = await attemptTranscription(audioBlob1);  // ← Waits here
  // HTTP request is IN FLIGHT (not blocking the thread)
}

// Your live recording (same time)
async function handleDoneClick() {
  const result = await attemptTranscription(liveBlob);  // ← Also waits here
  // Another HTTP request IN FLIGHT (both running simultaneously)
}
```

**Both functions wait for their own HTTP requests, but the requests happen in parallel!**

### 2. HTTP Requests to Deepgram API

Each `attemptTranscription()` call makes a **separate HTTP request**:

```typescript
// transcriptionRetry.ts (simplified)
export async function attemptTranscription(audioBlob, options) {
  for (let attempt = 1; attempt <= options.maxRapidAttempts; attempt++) {
    const response = await fetch('https://api.deepgram.com/v1/listen', {  // ← HTTP POST
      method: 'POST',
      headers: { 'Authorization': `Token ${apiKey}` },
      body: audioBlob,
    });
    
    const data = await response.json();
    return data.results.channels[0].alternatives[0].transcript;
  }
}
```

**Live recording call** → Makes HTTP request #1 to Deepgram  
**Pending clip call** → Makes HTTP request #2 to Deepgram  

**Deepgram API processes both requests in parallel on their servers!**

### 3. No Shared State Between Paths

**Live Recording State**:
- `recordNavState` (local to component)
- `audioBlob` (in memory)
- `currentClipId` (React state)

**Pending Clips State**:
- `isProcessingPending` (Zustand store)
- `audioRetrievalAttempts` (Zustand store)
- `pendingChildren` (queried from Zustand)

**Zero overlap!**
