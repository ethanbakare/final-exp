# 043_v6.7 SPINNER ANALYSIS AND FIX

**Date**: January 9, 2026
**Status**: 🔴 **ROOT CAUSE CONFIRMED**
**Issue**: Spinner not spinning during auto-retry (both home screen and record screen)

---

## 🎯 EXECUTIVE SUMMARY

**The Problem**: The spinner doesn't spin during auto-retry, even though the status correctly shows "Transcribing..."

**The Root Cause**: `activeHttpClipId` is **NEVER SET** in `processAllPendingClips` (auto-retry), but **IS SET** in manual retry.

**The Solution**: Two options:
1. **Option A (Simpler)**: Remove `isActiveRequest` logic entirely - make spinner spin whenever `status === 'transcribing'`
2. **Option B (Complex)**: Add `setActiveHttpClipId()` calls to `processAllPendingClips`

**Recommendation**: **Option A** - Remove `isActiveRequest`, simplify the code, and make spinner behavior match the mental model.

## 🤔 THE ARCHITECTURAL QUESTION

### Original Design Intent (Pre-v3)

**Before `'retry-pending'` status existed**, there was only:
- `'transcribing'` = could mean "actively making HTTP request" OR "waiting between retry intervals"

So `isActiveRequest` was needed to distinguish:
- `status='transcribing'` + `isActiveRequest=true` → Actively transcribing (spinner spins)
- `status='transcribing'` + `isActiveRequest=false` → Waiting between retries (spinner static)

### Current Design (Post-v3)

**Now we have distinct statuses**:
- `'transcribing'` = actively making HTTP request
- `'retry-pending'` = waiting between retries (moved to next parent)

**User's Valid Point**:
> "If `'transcribing'` ALWAYS means HTTP is happening, and `'retry-pending'` means we're waiting, then why do we need `isActiveRequest`? The status alone should be enough!"

## 🧩 WHAT HAPPENS DURING RETRY INTERVALS

### The Reality of `attemptTranscription`

**From transcriptionRetry.ts** - The retry loop:
```typescript
// Rapid attempts (3 attempts, immediate)
for (let i = 1; i <= maxRapidAttempts; i++) {
  const result = await transcribeRecording(blob, controller);  // ← HTTP happening (2-3s)
  if (success) return result;
}

// Interval waits
await sleep(30000);  // ← NO HTTP for 30 seconds
const result = await transcribeRecording(blob, controller);  // ← HTTP happening (2-3s)
if (success) return result;

await sleep(60000);  // ← NO HTTP for 1 minute
const result = await transcribeRecording(blob, controller);  // ← HTTP happening (2-3s)
if (success) return result;

await sleep(120000);  // ← NO HTTP for 2 minutes
const result = await transcribeRecording(blob, controller);  // ← HTTP happening (2-3s)
```

**Key Point**: Status stays `'transcribing'` throughout ALL of this (3-4 minutes total), but HTTP is only happening for ~15-20 seconds total.

**Question**: During the 30s, 1min, 2min sleep periods:
- Should the spinner spin? (indicating "working on it")
- Or should it be static? (indicating "waiting")

## 💡 TWO SOLUTIONS

### Option A: Remove `isActiveRequest` Logic (RECOMMENDED)

**Changes Required**:

1. **cliplist.tsx line 389-392**:
```typescript
// BEFORE:
<div className={`status-icon-wrapper ${isActiveRequest ? 'spinning-wrapper' : ''}`}>
  <svg className={`pending-icon ${isActiveRequest ? 'spinning-icon' : ''}`}

// AFTER:
<div className={`status-icon-wrapper ${status === 'transcribing' ? 'spinning-wrapper' : ''}`}>
  <svg className={`pending-icon ${status === 'transcribing' ? 'spinning-icon' : ''}`}
```

2. **ClipOffline.tsx line 115**:
```typescript
// BEFORE:
<TranscribeBig spinning={status === 'transcribing' && isActiveRequest !== false} />

// AFTER:
<TranscribeBig spinning={status === 'transcribing'} />
```

3. **Optional Cleanup** (if `activeHttpClipId` has no other uses):
   - Remove `activeHttpClipId` state from Zustand store
   - Remove `setActiveHttpClipId` calls from manual retry
   - Remove `activeHttpClipId` prop from ClipHomeScreen
   - Remove `isActiveRequest` derivation from getDisplayClip

**Pros**:
- ✅ Simpler code (less state to manage)
- ✅ Matches the mental model (status alone controls spinner)
- ✅ Fixes both home screen AND record screen
- ✅ No need to track HTTP granularly
- ✅ Spinner spins during waits (acceptable - shows "system is working")

**Cons**:
- ⚠️ Spinner spins during 30s, 1min, 2min interval waits
- ⚠️ User can't visually distinguish "active HTTP" from "waiting"

---

### Option B: Set `activeHttpClipId` in `processAllPendingClips`

**Changes Required**:

**ClipMasterScreen.tsx line 540** (add before `attemptTranscription`):
```typescript
// Set status to 'transcribing'
updateClip(firstClip.id, { status: 'transcribing' });

// ✅ ADD THIS: Track which clip is doing HTTP
setActiveHttpClipId(firstClip.id);

// ONE LOOP: Use shared retry logic
const result = await attemptTranscription(audioBlob, {
  maxRapidAttempts: 3,
  useIntervals: true,
  onProgress: (attempt, total) => {
    console.log(`[ProcessPending] Attempt ${attempt}/${total}`);
  }
});

// ✅ ADD THIS: Clear HTTP tracking
setActiveHttpClipId(null);
```

**The Problem with Option B**:
- `attemptTranscription` is ONE function call that takes 3-4 minutes
- From the outside, you can't see inside it
- `activeHttpClipId` would be set for the ENTIRE 3-4 minutes
- Spinner would spin CONTINUOUSLY, even during 30s, 1min, 2min waits
- This is the SAME END RESULT as Option A, but with more code

**To Truly Track Individual HTTP Requests** (not recommended):
Would require refactoring `attemptTranscription` to accept callbacks:
```typescript
await attemptTranscription(audioBlob, {
  onHttpStart: () => setActiveHttpClipId(id),  // ← Would need to add this
  onHttpEnd: () => setActiveHttpClipId(null),  // ← And this
});
```
This adds significant complexity for marginal benefit.

**Pros**:
- ✅ Keeps the existing architectural pattern
- ✅ Fixes both screens

**Cons**:
- ❌ More complex (need to remember to set/clear in both places)
- ❌ Same end result as Option A (spinner spins during waits)
- ❌ Misleading (activeHttpClipId set, but HTTP not always happening)

- User sees "Transcribing..." text (indicates work in progress)
