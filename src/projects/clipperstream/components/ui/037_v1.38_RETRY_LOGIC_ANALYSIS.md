## How Retry Logic Actually Works (Code Evidence)

### Current 3-Retry Mechanism

**File**: `useClipRecording.ts` (line 299-460)

**Flow**:
```
Attempt 1:
  → Send to /api/transcribe
  → Get error back from Deepgram
  → Classify error type (network/timeout/server-error)
  → If network: Retry immediately

Attempt 2:
  → Send to /api/transcribe again
  → Get error back
  → If network: Retry immediately

Attempt 3:
  → Send to /api/transcribe again
  → Get error back
  → If network: Enter interval mode (wait 1min, 2min, 4min, 5min)
  → Continue retrying indefinitely with intervals

Server errors (currently):
  → Don't retry at all
  → Return { text: '', error: 'server-error' }
```

**Key Code** (line 416-443):
```typescript
if (nextRetryCount < MAX_RAPID_ATTEMPTS) {  // 1-3
  // Rapid phase: immediate retry (no wait)
  retryTimerRef.current = setTimeout(() => transcribeRecording(), 0);
  return { text: '', error: 'network' };
} else {
  // Interval phase: wait before retry (1min, 2min, 4min, 5min)
  const waitTime = RETRY_INTERVALS[intervalIndex];
  retryTimerRef.current = setTimeout(() => transcribeRecording(), waitTime);
  return { text: '', error: 'network' };
}
```

**Critical discovery**: Each retry SENDS to Deepgram and GETS error back. NOT telling Deepgram to retry internally.

## The Critical Bug: 502/503/504 Errors

### Current Behavior (WRONG)

**Scenario**: Deepgram API is down

```
User finishes recording
  ↓
Send to Deepgram
  ↓
Deepgram returns HTTP 502 Bad Gateway
  ↓
Client receives: "Deepgram API error (HTTP 502): ..."
  ↓
Classified as 'server-error' (line 395-407 useClipRecording.ts)
  ↓
Does NOT retry ❌
  ↓
Returns { text: '', error: 'server-error' }
  ↓
ClipMasterScreen sees 'server-error'
  ↓
Shows error toast, NO pending clip created ❌
  ↓
Audio LOST ❌
```

**Why this is wrong**:
- 502/503/504 = Deepgram infrastructure down (not user's fault)
- Should be treated like network errors (retry + create pending clip)
- Currently treated like validation errors (don't retry, don't save)

---

### What Should Happen

**Scenario**: Deepgram API is down

```
User finishes recording
  ↓
Send to Deepgram
  ↓
Deepgram returns HTTP 502 Bad Gateway
  ↓
Classified as 'api-down' (NEW classification)
  ↓
circuitBreaker.recordFailure(error)
  ↓
Retry (attempt 2)
  ↓
Deepgram returns 502 again
  ↓
circuitBreaker.shouldUseWhisper() → YES
  ↓
Try OpenAI Whisper
  ↓
If Whisper succeeds → Use Whisper result ✅
If Whisper fails → Create pending clip (audio saved) ✅
```

## Edge Case: Bad Network But Shows "Online"

### User Concern

> "Network is so bad but there is still some network which shows online, not enough to transcribe. Person closes app. What happens?"

### Current Behavior

**Timeline**:
```
Recording done → Send to API → Waiting for response
  ↓
Network very slow (device shows "online")
  ↓
After 30 seconds: Timeout triggers (line 349 useClipRecording.ts)
  ↓
AbortController aborts request → AbortError
  ↓
Classified as 'network' → Retry (attempt 2)
  ↓
Another 30s timeout → Retry (attempt 3)
  ↓
After 3rd timeout (90 seconds total) → Creates pending clip ✅
```

**So**: Even if network is terrible, timeout ensures pending clip is created.

**But if user closes app before 90 seconds**:
- Audio already saved to IndexedDB (line 178 useClipRecording.ts) ✅
- Clip NOT created in Zustand store yet ❌
- Audio is orphaned in IndexedDB (can be recovered)

**Mitigation**: Unlikely user closes app during processing spinner.

## 400-Level Errors - Nearly Impossible

### Analysis

**400 Bad Request**: Nearly impossible
- Audio always recorded in WebM format
- Server validates before sending (line 140-170 transcribe.ts)
- Would require corrupted file (extremely rare)

**401 Unauthorized**: Impossible
- API key is server-side (env variable)
- Only fails if manually changed

**402 Insufficient Credits**: **Should handle specially** ✅
- Show specific error: "Deepgram account has insufficient credits"
- User needs to add funds

**415 Unsupported Media Type**: Impossible
- Always sends `audio/webm` MIME type
- Deepgram supports WebM

**429 Rate Limit**: Nearly impossible
- Retry logic has delays (1min, 2min, 4min, 5min)
- User can't spam requests

**Conclusion**: 400-level errors are edge cases. **500-level errors are critical to handle.**

## Circuit Breaker Integration - Architecture

### Recommended Approach: Option A with Separation

**Three separate files** (industry best practice):

#### 1. `transcriptionCircuitBreaker.ts` (NEW)
```typescript
/**
 * Circuit breaker for Deepgram service
 * Prevents repeated failures to slow API
 */
class TranscriptionCircuitBreaker {
  private state: 'CLOSED' | 'OPEN' | 'HALF_OPEN' = 'CLOSED';
  private failureCount: number = 0;

  shouldTryDeepgram(): boolean {
    return this.state === 'CLOSED' || this.state === 'HALF_OPEN';
  }

  shouldUseWhisper(): boolean {
    return this.state === 'OPEN';
  }

  recordFailure(error: Error): void {
    const isAPIDown =
      error.message.includes('502') ||
      error.message.includes('503') ||
      error.message.includes('504') ||
      error.message.includes('500');

    const isDNSError = error.message.includes('ENOTFOUND');

    if (isAPIDown || isDNSError) {
      this.failureCount++;
      if (this.failureCount >= 2) {
        this.state = 'OPEN';  // Switch to Whisper
      }
    }
  }

  recordSuccess(): void {
    this.state = 'CLOSED';
    this.failureCount = 0;
  }

  reset(): void {
    this.state = 'CLOSED';
    this.failureCount = 0;
  }
}

export const circuitBreaker = new TranscriptionCircuitBreaker();
```

**Responsibilities**:
- ✅ State management (CLOSED/OPEN/HALF_OPEN)
- ✅ Decision logic (should try Deepgram? should use Whisper?)
- ✅ Failure tracking

#### 2. `whisperProvider.ts` (NEW)
```typescript
/**
 * OpenAI Whisper transcription provider
 * Used as fallback when Deepgram fails
 */
export async function transcribeAudio(
  audioBlob: Blob,
  apiKey: string
): Promise<{ transcript: string; duration: number }> {
  const formData = new FormData();
  formData.append('file', audioBlob, 'recording.webm');
  formData.append('model', 'whisper-1');
  formData.append('response_format', 'verbose_json');

  const response = await fetch('https://api.openai.com/v1/audio/transcriptions', {
    method: 'POST',
    headers: {
      'Authorization': `Bearer ${apiKey}`,
    },
    body: formData,
  });

  if (!response.ok) {
    throw new Error(`Whisper API error (HTTP ${response.status})`);
  }

  const result = await response.json();

  return {
    transcript: result.text.trim(),
    duration: result.duration || 0,
  };
}
```

**Responsibilities**:
- ✅ Whisper API integration
- ✅ Same interface as Deepgram provider
- ✅ Error handling

```typescript
import { circuitBreaker } from '../api/transcriptionCircuitBreaker';
import { transcribeAudio as transcribeWithWhisper } from '../api/whisperProvider';

try {
  const response = await fetch('/api/clipperstream/transcribe', ...);
  const data = await response.json();

  // Success - record in circuit breaker
  circuitBreaker.recordSuccess();

  return { text: data.transcript, error: null };

} catch (error) {
  // Classify error type
  const isTimeout = error.name === 'AbortError';
  const isNetworkError = error instanceof TypeError;
  const isDNSError = error.message.includes('ENOTFOUND');

  // API down errors (NEW)
  const isAPIDown =
    error.message.includes('502') ||
    error.message.includes('503') ||
    error.message.includes('504') ||
    error.message.includes('500');

  // Client errors (don't retry)
  const isClientError =
    error.message.includes('400') ||
    error.message.includes('401') ||
    error.message.includes('402') ||
    error.message.includes('415');

  if (isClientError) {
    // Special handling for 402
    if (error.message.includes('402')) {
      return {
        text: '',
        error: 'validation',
        message: 'Deepgram account has insufficient credits. Please add funds.'
      };
    }
    return { text: '', error: 'server-error' };
  }

  // API down or DNS errors - try circuit breaker logic
  if (isAPIDown || isDNSError) {
    circuitBreaker.recordFailure(error);

    // If circuit breaker says use Whisper, try it now
    if (circuitBreaker.shouldUseWhisper()) {
      try {
        const whisperKey = process.env.NEXT_PUBLIC_OPENAI_API_KEY;
        if (!whisperKey) {
          console.warn('OpenAI API key not configured, cannot use Whisper fallback');
          // Continue with normal retry logic
        } else {
          console.log('🔄 Deepgram failed, trying OpenAI Whisper...');
          const whisperResult = await transcribeWithWhisper(blobToUse, whisperKey);
          circuitBreaker.recordSuccess();  // Whisper worked
          return { text: whisperResult.transcript, error: null };
        }
      } catch (whisperError) {
        console.error('Whisper fallback failed:', whisperError);
        // Continue with normal retry/pending clip logic
      }
    }

    // Either circuit not open yet, or Whisper failed
    // Continue with normal retry logic
    const shouldRetry = true;
  } else {
    // Timeout, network errors - normal retry logic
    const shouldRetry = isTimeout || isNetworkError;
  }

  if (shouldRetry) {
    const nextRetryCount = retryCount + 1;
    setRetryCount(nextRetryCount);

    if (nextRetryCount < MAX_RAPID_ATTEMPTS) {
      // Rapid retry (attempts 1-3)
      retryTimerRef.current = setTimeout(() => transcribeRecording(), 0);
      return { text: '', error: 'network' };
    } else {
      // Interval retry (attempts 4+)
      const waitTime = RETRY_INTERVALS[intervalIndex];
      retryTimerRef.current = setTimeout(() => transcribeRecording(), waitTime);
      return { text: '', error: 'network' };
    }
  }
}
```

- ✅ Retry logic (unchanged for timeout/network errors)

## Why This Architecture is Best

### Separation of Concerns

**Circuit Breaker** (transcriptionCircuitBreaker.ts):
- Knows: State management, failure thresholds
- Doesn't know: Retry logic, Whisper API

**Retry Logic** (useClipRecording.ts):
- Knows: When to retry, retry intervals
- Doesn't know: Circuit breaker internals, Whisper implementation

**Whisper Provider** (whisperProvider.ts):
- Knows: Whisper API integration
- Doesn't know: When to use it (circuit breaker decides)

### Benefits

✅ **Testable**: Each file can be tested independently
✅ **Maintainable**: Changes to circuit breaker don't affect retry logic
✅ **Reusable**: Circuit breaker can be used for other APIs if needed
✅ **Industry standard**: Clear separation of concerns

### NOT Option B (circuit breaker in transcribe.ts)

❌ Server-side state (multiple API instances = inconsistent state)
❌ Can't reuse for client-side decisions
❌ Violates separation (API route shouldn't decide fallback strategy)
