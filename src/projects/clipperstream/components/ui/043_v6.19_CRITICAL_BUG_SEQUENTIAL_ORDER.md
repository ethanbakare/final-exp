# CRITICAL BUG: Live Recording Breaks Sequential Order

**Date**: January 10, 2026  
**Severity**: 🔴 **CRITICAL - Breaks core UX feature**  
**Issue**: Live recordings append out of order when parent has pending clips  

---

## The Problem (User's Discovery)

### Scenario
- Parent File D has 5 pending clips (001, 002, 003, 004, 005)
- None transcribed yet (waiting in queue behind Parent A, B, C)
- You're viewing Parent D
- You press Record, speak, press Done
- **Live recording succeeds immediately**

### What Currently Happens (WRONG ❌)

**Step 1**: Live recording transcribes and appends (Line 902-908)
```
Parent D content: "Live recording text here..."  ← FIRST
```

**Step 2**: Later, auto-retry processes Parent D's pending clips
```
Clip 001 transcribes → Appends to parent
Parent D content: "Live recording text here... [Clip 001 text]"

Clip 002 transcribes → Appends to parent
Parent D content: "Live recording text here... [Clip 001 text] [Clip 002 text]"

... and so on
```

**Result**: 
```
Parent D content:
"[LIVE RECORDING - chronologically LAST]
 [Clip 001 - chronologically FIRST]
 [Clip 002 - chronologically SECOND]
 [Clip 003 - chronologically THIRD]
 [Clip 004 - chronologically FOURTH]
 [Clip 005 - chronologically FIFTH]"
```

**OUT OF ORDER! ❌**

---

### What Should Happen (CORRECT ✅)

**Step 1**: Check if parent has pending clips
```typescript
const pendingClipsInParent = clips.filter(c => 
  c.parentId === currentClipId && c.status === 'pending-child'
);

if (pendingClipsInParent.length > 0) {
  // ⛔ DO NOT transcribe live!
  // Create as pending clip instead
  handleOfflineRecording({ audioId, duration, currentClipId });
  return;
}
```

**Step 2**: Live recording becomes Clip 006 (pending)

**Step 3**: Auto-retry processes all clips in order
```
Clip 001 transcribes → Appends to parent
Clip 002 transcribes → Appends after 001
Clip 003 transcribes → Appends after 002
Clip 004 transcribes → Appends after 003
Clip 005 transcribes → Appends after 004
Clip 006 (live recording) transcribes → Appends after 005  ✅ CORRECT ORDER
```

**Result**:
```
Parent D content:
"[Clip 001 - chronologically FIRST]
 [Clip 002 - chronologically SECOND]
 [Clip 003 - chronologically THIRD]
 [Clip 004 - chronologically FOURTH]
 [Clip 005 - chronologically FIFTH]
 [Clip 006 - chronologically LAST]"  ✅ IN ORDER
```

## The Fix (Industry Best Practice)

### Approach: "Queue-Aware Live Recording"

**Principle**: Live recordings respect the pending clip queue for sequential order.

**Check Before Transcribing**:
1. Is this an append mode recording? (recording in existing file)
2. Does the parent have any pending clips?
3. If yes → Don't transcribe, become a pending clip
4. If no → Transcribe normally

```typescript
// 4. ✅ CHECK: Does parent have pending clips?
if (isAppendMode && currentClipId) {
  const clips = useClipStore.getState().clips;
  const pendingClipsInParent = clips.filter(c => 
    c.parentId === currentClipId && c.status === 'pending-child'
  );
  
  if (pendingClipsInParent.length > 0) {
    // ⛔ Parent has pending clips - preserve sequential order
    console.log('[HandleDone] Parent has pending clips, creating as pending instead of live transcription', {
      parentId: currentClipId,
      pendingCount: pendingClipsInParent.length
    });
    
    handleOfflineRecording({
      audioId: recordedAudioId!,
      duration: recordedDuration,
      currentClipId
    });
    
    // Show toast
    setShowAudioToast(true);
    
    // Stay in current state (parent has content or pending clips)
    const currentClip = getClipById(currentClipId);
    const hasContent = currentClip && currentClip.content && currentClip.content.length > 0;
    setRecordNavState(hasContent ? 'complete' : 'record');
    
    return;  // ← Exit early, don't transcribe
  }
}

// 5. No pending clips → Transcribe normally
const result = await attemptTranscription(recordedBlob, {
  maxRapidAttempts: 3,
  useIntervals: false,
});

const { text: rawText, error: transcriptionError } = result;
```
