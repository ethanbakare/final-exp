# 037_v1.5 - FIX: Preserve Blob MIME Type for Offline Recordings

**Date**: January 1, 2026
**Type**: 🔧 CRITICAL FIX
**Status**: 📋 READY TO APPLY
**Related**: See 037_v1.4 for root cause analysis

---

## Problem Summary

Offline-recorded clips fail with 500 error because Blob MIME type is lost when retrieved from IndexedDB.

**Flow**:
1. Record offline → Blob stored with `type: 'audio/webm;codecs=opus'`
2. Go online → Blob retrieved with `type: ''` (EMPTY!)
3. API receives file with no MIME type → Deepgram fails → 500 error

---

## Solution: Store MIME Type in Clip Metadata

Instead of relying on IndexedDB to preserve Blob.type, store the MIME type in the Clip object.

---

## Implementation

### Step 1: Add mimeType to Clip Interface

**Location**: [clipTypes.ts](clipTypes.ts) - Clip interface

**ADD** this field to the Clip interface:

### Step 3: Recreate Blob with Correct MIME Type Before Transcription

**Location**: [ClipMasterScreen.tsx:998-1018](ClipMasterScreen.tsx#L998-L1018) - processChild function

**FIND** (Lines 996-1018):
```typescript
try {
  // Step 1: Get audio from IndexedDB
  const audioBlob = await getAudio(child.audioId!);
  if (!audioBlob) {
    console.warn('[ProcessChild] Audio not found for:', child.id);
    updateClip(child.id, {
      status: 'failed',
      transcriptionError: 'Audio not found in storage'
    });
    return { success: false, rawText: '', formattedText: '' };
  }

  // Step 2: Update status to transcribing
  updateClip(child.id, { status: 'transcribing' });

  // Step 3: Transcribe (uses existing retry mechanism)
  const transcriptionResult = await transcribeRecording(audioBlob);
  const { text: rawText, error: transcriptionError } = transcriptionResult;
```

**REPLACE WITH**:
```typescript
try {
  // Step 1: Get audio from IndexedDB
  const audioBlob = await getAudio(child.audioId!);
  if (!audioBlob) {
    console.warn('[ProcessChild] Audio not found for:', child.id);
    updateClip(child.id, {
      status: 'failed',
      transcriptionError: 'Audio not found in storage'
    });
    return { success: false, rawText: '', formattedText: '' };
  }

  // CRITICAL FIX: Recreate blob with correct MIME type if lost during IndexedDB round-trip
  // IndexedDB sometimes loses Blob.type property, causing API to reject audio
  const mimeType = child.mimeType || 'audio/webm';  // Get stored MIME type from clip metadata
  const correctedBlob = audioBlob.type === ''
    ? new Blob([audioBlob], { type: mimeType })  // Recreate with correct type
    : audioBlob;  // Use original if type was preserved

  console.log('[ProcessChild] Audio retrieved:', {
    id: child.id,
    size: audioBlob.size,
    originalType: audioBlob.type,
    storedMimeType: child.mimeType,
    finalType: correctedBlob.type,
    wasRecreated: audioBlob.type === ''
  });

  // Step 2: Update status to transcribing
  updateClip(child.id, { status: 'transcribing' });

  // Step 3: Transcribe (uses existing retry mechanism)
  const transcriptionResult = await transcribeRecording(correctedBlob);
  const { text: rawText, error: transcriptionError } = transcriptionResult;
```

## How This Fix Works

### Before Fix
```
1. Store blob with type 'audio/webm;codecs=opus'
2. IndexedDB serializes blob → type becomes ''
3. Retrieve blob with type '' (EMPTY)
4. Send to API: FormData creates File with type ''
5. API receives mimetype: '' or undefined
6. Deepgram rejects: 500 error ❌
```

### After Fix
```
1. Store blob + mimeType field in Clip object
2. IndexedDB serializes blob → type becomes ''
3. Retrieve blob with type '' + mimeType from Clip
4. Recreate: new Blob([blob], { type: mimeType })
5. Send to API: FormData creates File with type 'audio/webm;codecs=opus'
6. API receives mimetype: 'audio/webm;codecs=opus'
7. Deepgram transcribes successfully ✅
```

## Why This Approach is Better

**Alternative approaches considered**:
1. ❌ Modify audioStorage.ts to store MIME type separately

   - Requires migrating existing IndexedDB data

2. ✅ **Store MIME type in Clip object** (THIS APPROACH)
   - Simple, minimal changes
   - No storage layer changes needed
   - MIME type lives with the clip metadata (makes sense conceptually)
