# v1.52 UI Updates Summary

## Overview
This document outlines all UI component changes made for VPN detection and improved retry status feedback. These changes are part of **v1.50 implementation**, which pivoted away from a Whisper fallback strategy after testing revealed that VPNs block both Deepgram AND Whisper APIs.

## Context: v1.50 Strategy Change

### What Happened:
1. **Original Plan (v1.38-v1.40)**: Use Whisper as fallback when Deepgram fails due to DNS/VPN issues
2. **VPN Test Results (Phase 0.5)**: Whisper ALSO fails with `ENOTFOUND api.openai.com` when VPN is active
3. **New Strategy (v1.50)**: 
   - DNS/network errors do NOT trigger circuit breaker or provider switching
   - Instead, show persistent VPN warning toast to inform user
   - Circuit breaker ONLY applies to API-specific errors (401, 402, 429, 500-series)
   - User must disable VPN to allow transcription to succeed

### UI Requirements:
These UI components support the v1.50 strategy by clearly communicating VPN blocking to users and distinguishing it from other failure types.

**Purpose:** Persistent notification to inform users that VPN is blocking transcription services

**Persistent Behavior (Planned):**
- User can dismiss the toast by clicking the X button
- If DNS errors continue, toast automatically re-appears after 30 seconds
- Toast does NOT prevent background retry attempts (system keeps trying)
- Toast persists across screen navigation until VPN is disabled or dismissed

**Implementation Logic (Planned for v1.50):**
```typescript
// ClipMasterScreen.tsx
const [showVpnToast, setShowVpnToast] = useState(false);
const [vpnToastDismissedAt, setVpnToastDismissedAt] = useState<number | null>(null);

// Auto-reshow logic
useEffect(() => {
  if (!showVpnToast && vpnToastDismissedAt) {
    const timeSinceDismiss = Date.now() - vpnToastDismissedAt;
    if (timeSinceDismiss > 30000 && hasPendingDnsErrors()) {
      setShowVpnToast(true); // Reshow after 30s if errors persist
    }
  }
}, [showVpnToast, vpnToastDismissedAt, clips]);

// Dismiss handler
const handleVpnToastClose = () => {
  setShowVpnToast(false);
  setVpnToastDismissedAt(Date.now());
};
```

## Error Flow: From Detection to UI Display

### Step 1: Error Detection
**File:** `useClipRecording.ts` or `ClipMasterScreen.tsx`

```typescript
// During transcription
try {
  await transcribe(audioBlob);
} catch (error) {
  // Check error type
  if (error.message.includes('ENOTFOUND') || error.message.includes('ECONNREFUSED')) {
    return { success: false, error: 'dns-block' }; // VPN detected
  }
  // Other errors...
}
```

### Step 2: Error Handling
**File:** `ClipMasterScreen.tsx`

```typescript
// On transcription failure
if (result.error === 'dns-block') {
  // UI Updates
  setShowVpnToast(true); // Show persistent warning
  
  // State Updates
  updateClip(clipId, {
    status: 'pending-retry', // Keeps retrying
    lastError: 'dns-block'   // Marks as VPN issue
  });
  
  // Circuit breaker: NOT triggered for DNS errors
  // Background retries: CONTINUE
}
```

### Step 3: UI Display
**Files:** `ClipList.tsx`, `ClipOffline.tsx`, `ClipMasterScreen.tsx`

```typescript
// Status determination
const displayStatus = clip.lastError === 'dns-block' 
  ? 'vpn-blocked'      // Orange "Blocked by VPN" + warning icon
  : 'retry-pending';   // White "Retrying soon..." + static spinner

// Toast display
{showVpnToast && <VpnToast onClose={...} />}
```

**Result:**
- User sees VPN toast (dismissible)
- Clip shows "Blocked by VPN" status
- System continues retrying in background
- Toast re-appears if issue persists after 30s

## Implementation Notes

### Design Patterns:
- All components follow existing design patterns (opacity crossfades, slide-in animations, etc.)
- Icon sizes are consistent (24×24px for buttons, 12×12px for inline status icons)
- Color scheme uses existing CSS variables where possible
- All new states are backward-compatible (don't break existing usage)

### Key Architectural Decisions (v1.50):
1. **No Provider Switching**: DNS errors do not trigger Whisper fallback (VPN blocks both)
2. **Persistent User Notification**: VpnToast is dismissible but re-shows if issue persists
3. **Background Retries Continue**: DNS errors don't stop retry attempts, only inform user
4. **Circuit Breaker Exclusion**: DNS/network errors bypass circuit breaker logic
5. **Visual Distinction**: `'vpn-blocked'` has unique orange color to differentiate from other failures

### v1.50 Connection:
- These UI components are the **presentation layer** for v1.50 error handling strategy
- They provide clear user feedback when VPN blocks transcription APIs
- They support the "inform and continue retrying" approach (vs. "fail and switch providers")
