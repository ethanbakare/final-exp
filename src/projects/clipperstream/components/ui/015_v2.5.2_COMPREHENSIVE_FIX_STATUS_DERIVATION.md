## 🐛 PROBLEMS FROM v2.5.1

### Problem #1: Children Appearing as "Duplicates" on Home Screen

**What the user saw:**
- Started with home screen showing 3 parent clips (Recording 01, 02, 03)
- Went online → Auto-retry started
- Suddenly 6 "new" clips appeared on home screen
- These weren't duplicates - they were **children becoming visible**

**Root Cause:**
```typescript
// ClipMasterScreen.tsx line 1152 (BUGGY)
const homeScreenClips = useMemo(() => {
  return clips.filter(clip => clip.status !== 'pending-child');
}, [clips]);
```

**Why it breaks:**
1. Filter checks `status !== 'pending-child'` to hide children
2. When auto-retry starts, it changes child status from `'pending-child'` → `'transcribing'`
3. Now children pass the filter (`'transcribing' !== 'pending-child'`) → they appear on home screen
4. User sees 6 "duplicates" (actually 6 children that should stay hidden)

**Evidence from console:**
```

[INFO] Status transition { clipId: "child-1", from: "pending-child", to: "transcribing" }
[INFO] Status transition { clipId: "child-2", from: "pending-child", to: "transcribing" }
...
```

**After these transitions, all 6 children appear on home screen because status is now 'transcribing'.**

## ✅ THE COMPREHENSIVE FIX

### Fix #1: Filter Children by parentId (Not by Status)

**File:** [ClipMasterScreen.tsx:1152](ClipMasterScreen.tsx#L1152)

**BEFORE (v2.5.1 - BUGGY):**
```typescript
const homeScreenClips = useMemo(() => {
  return clips.filter(clip => clip.status !== 'pending-child');
}, [clips]);
```

**AFTER (v2.5.2 - CORRECT):**
```typescript
// v2.5.2 FIX: Filter by parentId, not status
// Children ALWAYS have parentId, regardless of status transitions
const homeScreenClips = useMemo(() => {
  return clips.filter(clip => !clip.parentId);  // Only show parents (no parentId)
}, [clips]);
```

**Why this works:**
- Parents: `parentId: undefined` → pass filter (shown on home)
- Children: `parentId: "parent-id"` → blocked by filter (hidden from home)
- Status can change ('pending-child' → 'transcribing' → null) but parentId never changes
- Children stay hidden regardless of status transitions

**Result:**
- ✅ 6 children stay hidden when auto-retry starts
- ✅ Only 3 parent clips shown on home screen
- ✅ No more "duplicate" clips appearing

### Fix #3: Derive Complete Status in ClipHomeScreen

**File:** [ClipHomeScreen.tsx:137-152](ClipHomeScreen.tsx#L137-L152)

**BEFORE (v2.5.1 - Only derives status, not isActiveRequest):**
```typescript
const getDisplayClip = useCallback((clip: Clip): Clip => {
  const allClips = getClips();
  const children = allClips.filter(c => c.parentId === clip.id);
  const hasPendingChildren = children.some(c =>
    c.status === 'pending-child' || c.status === 'transcribing'
  );

    return { ...clip, status: 'pending' as const };

  return clip;
}, []);
```

**AFTER (v2.5.2 - Derives both status AND isActiveRequest):**
```typescript
// v2.5.2 FIX: Derive both status AND isActiveRequest from children
// This creates the complete state system: pending/transcribing × active/waiting
const getDisplayClip = useCallback((
  clip: Clip,
  allClips: Clip[],
  activeTranscriptionClipId: string | null,
  isActiveRequest: boolean
): Clip => {
  // Find children for this parent
  const children = allClips.filter(c => c.parentId === clip.id);

  if (children.length === 0) {
    // No children - show clip as-is
    return clip;
  }

  // Derive status from children's states
  const hasTranscribingChildren = children.some(c => c.status === 'transcribing');
  const hasPendingChildren = children.some(c => c.status === 'pending-child');

  let derivedStatus: Clip['status'] = clip.status;

  if (hasTranscribingChildren) {
    // At least one child is transcribing
    derivedStatus = 'transcribing';
  } else if (hasPendingChildren) {
    // Children waiting to transcribe
    derivedStatus = 'pending';
  }

  // Derive isActiveRequest: only true if one of this parent's children is actively being transcribed
  const derivedIsActiveRequest =
    activeTranscriptionClipId !== null &&
    children.some(c => c.id === activeTranscriptionClipId) &&
    isActiveRequest;  // From useClipRecording (true during HTTP, false during waits)

  return {
    ...clip,
    status: derivedStatus,
    isActiveRequest: derivedIsActiveRequest
  };
}, []);
```

## 📊 STATE TRANSITIONS

**Offline recording flow:**

```
1. Create parent (status: null) + child (status: 'pending-child')
   → Parent shows: "Waiting to transcribe" (static ⟳)

2. Go online → Auto-retry starts for child
   → setActiveTranscriptionClipId(child.id)
   → Child status: 'pending-child' → 'transcribing'
   → isActiveRequest: true (HTTP active)
   → Parent shows: "Transcribing..." (spinning ↻)

3. During interval wait (e.g., 1min wait)
   → isActiveRequest: false
   → Child still has status: 'transcribing'
   → Parent shows: "Transcribing..." (static ⟳)

4. HTTP resumes
   → isActiveRequest: true
   → Parent shows: "Transcribing..." (spinning ↻)

5. Transcription succeeds
   → setActiveTranscriptionClipId(null)
   → Child status: 'transcribing' → null
   → Parent shows: (no status)
```

## 📌 ARCHITECTURE NOTES

### Why Filter by parentId Instead of Status?

**Problem with status-based filtering:**
- Children go through status transitions: `'pending-child'` → `'transcribing'` → `null`
- Each transition could break the filter
- Fragile: any new status would need filter updates

**Advantage of parentId-based filtering:**
- `parentId` is set once when clip is created, never changes
- Survives all status transitions
- Clear semantic meaning: "children have parents, parents don't"
- Robust: works for any future status values

### Why Track activeTranscriptionClipId in ClipMasterScreen?

**Alternative considered:** Add `isActiveRequest` to Clip storage

**Why we don't:**
- `isActiveRequest` is transient UI state, not data
- Pollutes storage with display logic
- Would need constant updates during transcription
- Not relevant when app restarts (stale data)

**Why activeTranscriptionClipId in component state:**
- Lives only during session (appropriate for transient state)
- ClipMasterScreen already orchestrates transcription
- Can be derived and passed down to ClipHomeScreen
- Doesn't pollute storage schema

### Why Derive in ClipHomeScreen Instead of ClipMasterScreen?

**Could derive in ClipMasterScreen and pass pre-computed displayClips:**
```typescript
const displayClips = homeScreenClips.map(clip =>
  deriveStatus(clip, allClips, activeTranscriptionClipId, isActiveRequest)
);
```

**Why we derive in ClipHomeScreen instead:**
- ClipHomeScreen already has `getDisplayClip` helper (from v2.5.1)
- Keeps derivation logic close to where it's used (ClipListItem)
- ClipHomeScreen can call `getClips()` to get all clips for child lookup
- Cleaner separation: ClipMasterScreen manages state, ClipHomeScreen manages display

### State Management Philosophy

**ClipMasterScreen (Orchestrator):**
- Owns transcription coordination
- Tracks which clip is actively being transcribed
- Exposes raw state (`activeTranscriptionClipId`, `isActiveRequest`)

**ClipHomeScreen (Presenter):**
- Derives display state from raw state + clip data
- Computes what user should see
- Passes computed props to ClipListItem

**ClipListItem (View):**
- Pure presentation component
- Renders based on props (status + isActiveRequest)
- No business logic, just visual states

This follows the **Container/Presenter pattern** common in React architecture.

## 🔄 COMPARISON TO ClipOfflineScreen.tsx

ClipOfflineScreen demonstrates the state system we're implementing for production:

**ClipOfflineScreen (Reference):**
```typescript
type OfflineState = 'waiting' | 'attemptActive' | 'betweenAttempts' | 'done';

const getClipStatus = (state: OfflineState): 'pending' | 'transcribing' | null => {
  switch (state) {
    case 'waiting': return 'pending';
    case 'attemptActive':
    case 'betweenAttempts': return 'transcribing';
    case 'done': return null;
  }
};

const getIsActiveRequest = (state: OfflineState): boolean => {
  return state === 'attemptActive';
};
```

**Production (v2.5.2):**
```typescript
// Derive status from children's statuses
const hasTranscribingChildren = children.some(c => c.status === 'transcribing');
const hasPendingChildren = children.some(c => c.status === 'pending-child');

let derivedStatus: Clip['status'] = clip.status;
if (hasTranscribingChildren) derivedStatus = 'transcribing';
else if (hasPendingChildren) derivedStatus = 'pending';

// Derive isActiveRequest from active transcription tracking
const derivedIsActiveRequest =
  activeTranscriptionClipId !== null &&
  children.some(c => c.id === activeTranscriptionClipId) &&
  isActiveRequest;
```

**Mapping:**
- `waiting` → `status: 'pending'`, `isActiveRequest: false`
- `attemptActive` → `status: 'transcribing'`, `isActiveRequest: true`
- `betweenAttempts` → `status: 'transcribing'`, `isActiveRequest: false`
- `done` → `status: null`, `isActiveRequest: false`

## 💡 EDGE CASES HANDLED

### Edge Case #1: Auto-Retry Fails Mid-Transcription

**Scenario:**
- Auto-retry starts transcribing child-1
- Network drops during HTTP request
- `setActiveTranscriptionClipId(null)` not called

**Fix:** Add try/catch to ensure cleanup:
```typescript
try {
  setActiveTranscriptionClipId(clip.id);
  await transcribeRecording(audioBlob);
} catch (error) {
  // ... error handling
} finally {
  setActiveTranscriptionClipId(null);  // Always clear
}
```

---

### Edge Case #2: User Navigates Away During Transcription

**Scenario:**
- Auto-retry transcribing in background
- User navigates to different screen
- Returns to home screen

**Behavior:**
- `activeTranscriptionClipId` persists in ClipMasterScreen state
- Home screen re-derives status using current `activeTranscriptionClipId`
- Spinner shows correctly (no visual glitch)

**Why it works:**
- State lives in ClipMasterScreen (persists across screen nav)
- ClipHomeScreen derives on every render (always fresh)

---

### Edge Case #3: Multiple Rapid Status Changes

**Scenario:**
- Auto-retry transcribing 6 children sequentially
- Each completes in <1 second
- Rapid: `setActiveTranscriptionClipId(child-1)` → `null` → `child-2` → `null` → ...

**Behavior:**
- React batches rapid state updates
- Parent clip's spinner toggles on/off quickly
- May appear as continuous spinning (acceptable)

**Why acceptable:**
- Visually smooth (React batching prevents jank)
- Semantically correct (active clip is tracked)
- Edge case (most transcriptions take >1 second)
