## Issue #1: NavBar State Not Updating After Background Transcription

### Problem Statement

**User Experience**:
```
SCENARIO:
1. User records offline → Clip becomes pending (shows ClipOffline UI)
2. User clicks Done → Nav bar shows 'record' state (just Record button)
3. User goes online → Auto-retry starts transcribing in background
4. User navigates to parent clip's record screen while transcription happening
5. Transcription completes successfully → text appears on screen
6. ❌ Nav bar STAYS in 'record' state (only Record button visible)
7. ❌ Copy and Structure buttons DO NOT APPEAR
8. ❌ User cannot copy transcribed text despite it being visible
```

**Expected Behavior**:
- When transcription completes for the currently viewed clip, nav bar should transition from 'record' → 'complete' state
- Copy and Structure buttons should fade in with opacity animation (same as navigating to an existing clip from home screen)
- Should match behavior of navigating to a transcribed clip from home screen

### Root Cause Analysis

#### Current Flow (Broken)

**File**: `ClipMasterScreen.tsx`

**Offline Recording** (lines 801-812):
```typescript

```

**Key insight**: When recording offline, nav bar goes to **'record' state** (NOT 'processing' state) because there's no active transcription happening.

**How it works**:

### Architecture Flaw

The disconnect stems from **separation of concerns that went too far**:

```
Live Recording Path:
  handleDoneClick 
    → formatTranscriptionInBackground 
      → Updates clip content
      → ✅ Updates nav bar state (if viewing)

Background Retry Path:
  processAllPendingClips
    → formatChildTranscription
      → Updates parent content
      → ❌ Does NOT update nav bar state
```

**Why this happened**:
1. `formatTranscriptionInBackground` was designed for **live recordings**
2. `processAllPendingClips` was added later for **auto-retry**
3. Auto-retry uses a different formatting function (`formatChildTranscription`)
4. **Nobody connected** auto-retry back to nav bar state

---

### The Fix

**Location**: `ClipMasterScreen.tsx`, line 611 (after `updateClip` in `processAllPendingClips`)

**Add this logic**:
```typescript
// Update parent
const updatedContent = {
  rawText: currentParent.rawText
    ? currentParent.rawText + '\n\n' + result.text
    : result.text,
  formattedText: currentParent.formattedText
    ? currentParent.formattedText + ' ' + formattedText
    : formattedText,
  content: currentParent.formattedText
    ? currentParent.formattedText + ' ' + formattedText
    : formattedText,
};

updateClip(currentParent.id, updatedContent);

// ✅ NEW: Update nav bar if viewing this parent clip
if (currentClipId === currentParent.id) {
  console.log('[ProcessPending] Transcription complete for current clip, updating nav bar');
  setRecordNavState('complete');
}

// Delete child and audio
deleteClip(firstClip.id);
```

**Why this works**:
1. `currentClipId` tracks which clip the user is viewing
2. `currentParent.id` is the parent being updated
3. If they match, user is viewing this clip's record screen
4. Transition nav bar to 'complete' to show copy/structure buttons

**User clarification about flickering**:
> "I don't want situation whereby you're on the home screen or you're on a different screen and those buttons are being triggered. You don't want the buttons responding on the home screen because we also have the record button showing on the home screen. it would be weird to have it going back and forth or flickering."

**This fix addresses that**:
- Only updates nav bar if `currentClipId === currentParent.id`
- If user is on home screen, `currentClipId` would be different (or null)
- No flickering, only updates when actually viewing the clip

## Root Cause Summary

Both issues stem from the same architectural gap:

**Gap**: Auto-retry system (`processAllPendingClips`) operates independently from UI state management (`recordNavState`, toast triggers).

**Why it happened**:
1. Auto-retry added as separate system
2. Focused on data/network layer (correct!)
3. **Forgot to wire back to UI layer** (oversight!)

**State Flow Clarification**:
- **Offline recording**: 'record' state (no transcription happening yet)
- **Live recording**: 'record' → 'recording' → 'processing' → 'complete' (full flow)
- **Background transcription completes**: Should transition 'record' → 'complete' (simple fade)

**Animation Used**:
- NOT 'morph' animation (no expansion, no processing spinner)
- Uses 'fade' animation (simple opacity crossfade of buttons)
- Same as navigating from home screen to existing transcribed clip

**Lesson learned**:
> "When adding background processes that affect user-visible data, **always check**: Does the UI need to know when this process completes?"
