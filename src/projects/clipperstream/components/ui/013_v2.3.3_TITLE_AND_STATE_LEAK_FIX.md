## 🐛 Bug Reports from Testing

### Bug #1: Children Have Wrong Title

**User report:**
> "Recording 01 shows 4 clips (001-004) in UI, but session storage shows 3 clips with title 'Recording 02' instead of 'Recording 01'!"

**Evidence from storage:**
```javascript
clips = [
  { id: "...", title: "Recording 03", status: null },      // Parent for Rec 03
  { id: "...", title: "Recording 02", status: null },      // Parent for Rec 02
  { id: "...", title: "Recording 02", status: null },      // ← Child (WRONG!)
  { id: "...", title: "Recording 02", status: null },      // ← Child (WRONG!)
  { id: "...", title: "Recording 02", status: null },      // ← Child (WRONG!)
  { id: "...", title: "Recording 01" },                    // Parent for Rec 01
  { id: "...", title: "Mary's Unusual Tale of a Lamb" }    // Transcribed
]
```

**Expected:**
- Children of "Recording 01" should have title: "Recording 01"
- Not title: "Recording 02"

**Root cause:**

In v2.3.1, when creating a child, we call:

```typescript
const childClip = createNewClip(
  '',                   // content
  parentClip.title,     // title ← Should copy parent's title
  ''                    // summary
);
```

But `createNewClip()` has this signature:
```typescript
function createNewClip(content: string, recordingNumber: string, summary: string)
```

The second parameter is `recordingNumber`, which gets passed to `getNextRecordingNumber()` internally!

So when we pass `parentClip.title = "Recording 01"`:
- `createNewClip()` ignores it
- Calls `getNextRecordingNumber()` instead
- Finds 1 existing parent → Returns "Recording 02"
- Child gets wrong title!

### Bug #2: Transcript Appears in Wrong Recording

**User report:**
> "After Recording 03 finished transcribing, I clicked into Recording 01. Recording 03's transcript appeared above the 4 clips! When I navigated out and back in, it disappeared."

**Evidence from console:**
```
Line 107: handleOnline FIRED - network is back online
Line 110: Found pending clips for auto-retry
Line 111: Status transition (Recording 03 → transcribing)
Line 116: Auto-retrying transcription
Line 118: Loaded parent with children ← User clicked Recording 01
Line 119: Transcription successful ← Recording 03 finishes
Line 120: Status transition (Recording 03 → transcribed)
Line 124: Background transcription completed
Line 130: Navigated to home screen
Line 141: Loaded parent with children ← User enters Recording 01 again
```

**What's happening:**

1. Recording 03 is transcribing in background
2. User clicks into Recording 01
3. `handleClipClick` loads Recording 01's clips
4. Recording 03's transcription completes → triggers state update
5. React re-renders Recording 01's screen
6. But `currentClipId` briefly points to Recording 03
7. Recording 03's transcript appears in Recording 01's UI
8. When navigating out/in, state resets correctly

**Root cause:**

`currentClipId` is being updated by background transcription, but the UI doesn't check if it's still viewing the same clip.

## ✅ THE FIXES

### Fix #1: Children Should NOT Call `createNewClip()`

**File:** `ClipMasterScreen.tsx`
**Location:** Offline handler, child creation logic (line ~1280-1320)

**Problem:**

`createNewClip()` generates new recording numbers, but children should use parent's title.

**Solution:**

Don't use `createNewClip()` for children. Create the child clip object directly.

**BEFORE (v2.3.1 - Uses createNewClip):**

```typescript
} else {
  // Create CHILD recording linked to parent
  const parentClip = clips.find(c => c.id === currentClipId);

  // Get next pending clip number for this parent
  const existingChildClips = clips.filter(c => c.parentId === currentClipId);
  const childNumbers = existingChildClips
    .map(c => c.pendingClipTitle?.match(/Clip (\d+)/)?.[1])
    .filter(n => n)
    .map(Number);
  const maxChildNumber = childNumbers.length > 0 ? Math.max(...childNumbers) : 0;
  const nextChildNumber = String(maxChildNumber + 1).padStart(3, '0');
  const nextPendingTitle = `Clip ${nextChildNumber}`;

  // Create child clip (BUGGY - uses createNewClip)
  const childId = `clip-${Date.now()}-${Math.random().toString(36).substring(2, 15)}`;

  const childClip = createNewClip(
    '',  // No content yet
    parentClip.title,  // ← BUG: createNewClip ignores this, generates "Recording 02"
    ''   // No summary
  );

  // Add to selectedPendingClips array
  setSelectedPendingClips(prev => [
    ...prev,
    { id: childId, title: nextPendingTitle }
  ]);

  log.info('Created CHILD for offline recording', {
    childId: childId,
    parentId: currentClipId,
    parentStatus: parentClip.status,
    title: nextPendingTitle
  });

  // Update storage
  updateClip(childId, {
    status: 'pending-child',
    pendingClipTitle: nextPendingTitle,
    audioId: audioId,
    parentId: currentClipId
  });
}
```

**AFTER (v2.3.3 - Create child directly):**

```typescript
} else {
  // Create CHILD recording linked to parent
  const parentClip = clips.find(c => c.id === currentClipId);

  if (!parentClip) {
    log.error('Parent clip not found, cannot create child', {
      currentClipId
    });
    return;
  }

  // Get next pending clip number for this parent
  const existingChildClips = clips.filter(c => c.parentId === currentClipId);
  const childNumbers = existingChildClips
    .map(c => c.pendingClipTitle?.match(/Clip (\d+)/)?.[1])
    .filter(n => n)
    .map(Number);
  const maxChildNumber = childNumbers.length > 0 ? Math.max(...childNumbers) : 0;
  const nextChildNumber = String(maxChildNumber + 1).padStart(3, '0');
  const nextPendingTitle = `Clip ${nextChildNumber}`;

  // v2.3.3 FIX: Create child clip DIRECTLY (don't use createNewClip)
  // This preserves parent's title instead of generating new recording number
  const childId = `clip-${Date.now()}-${Math.random().toString(36).substring(2, 15)}`;

  const now = new Date();
  const childClip: Clip = {
    id: childId,
    title: parentClip.title,            // ← CORRECT: Use parent's title
    date: now.toLocaleDateString('en-US', { month: 'short', day: 'numeric', year: 'numeric' }),
    content: '',                        // No content yet (pending transcription)
    summary: '',                        // No summary
    pendingClipTitle: nextPendingTitle, // "Clip 002", "Clip 003", etc.
    audioId: audioId,                   // Link to IndexedDB audio
    status: 'pending-child',            // Child status (filtered from home)
    parentId: currentClipId             // Link to parent
  };

  // Save to storage
  updateClip(childClip.id, childClip);

  // Add to selectedPendingClips array
  setSelectedPendingClips(prev => [
    ...prev,
    { id: childId, title: nextPendingTitle }
  ]);

  log.info('Created CHILD for offline recording', {
    childId: childId,
    parentId: currentClipId,
    parentTitle: parentClip.title,  // Log parent title for debugging
    parentStatus: parentClip.status,
    childTitle: nextPendingTitle
  });
}
```

**Why this works:**

```
Before fix:
- createNewClip('', 'Recording 01', '')
  → Ignores 'Recording 01'
  → Calls getNextRecordingNumber()
  → Returns 'Recording 02'
  → Child has title: 'Recording 02' ❌

After fix:
- Create clip object directly
- Set title: parentClip.title ('Recording 01')
- Child has title: 'Recording 01' ✅
```

### Fix #2: Prevent State Leak During Background Transcription

**File:** `ClipMasterScreen.tsx`
**Location:** Transcription success handler (line ~850-900)

**Problem:**

When background transcription completes, it updates `currentClipId` to the transcribed clip, even if user is viewing a different clip.

**Solution:**

Don't change `currentClipId` during background transcription. Only update the clip's content in storage.

**BEFORE (v2.3.1 - Updates currentClipId):**

```typescript
const handleTranscriptionSuccess = useCallback((clipId: string, transcription: string) => {
  log.info('Transcription successful', { clipId });

  // Update current clip ID (BUGGY - changes while user viewing different clip)
  setCurrentClipId(clipId);

  // Update clip content
  updateClip(clipId, {
    content: transcription,
    status: 'transcribed'
  });

  // Start background formatting
  formatText(transcription, clipId);
}, [updateClip, formatText]);
```

**AFTER (v2.3.3 - Don't update currentClipId):**

```typescript
const handleTranscriptionSuccess = useCallback((clipId: string, transcription: string) => {
  log.info('Transcription successful', { clipId });

  // v2.3.3 FIX: Don't update currentClipId during background transcription
  // Only update storage - let UI refresh from storage
  // This prevents transcript from leaking into wrong recording when user navigates

  // Update clip content in storage
  updateClip(clipId, {
    content: transcription,
    status: 'transcribed'
  });

  log.debug('Background transcription completed', {
    clipId,
    transcriptionLength: transcription.length,
    note: 'currentClipId unchanged (prevents state leak)'
  });

  // Start background formatting
  formatText(transcription, clipId);
}, [updateClip, formatText]);
```

**Alternative Fix (If Above Doesn't Work):**

If background transcription needs to update `currentClipId`, add a check:

```typescript
const handleTranscriptionSuccess = useCallback((clipId: string, transcription: string) => {
  log.info('Transcription successful', { clipId });

  // v2.3.3 FIX: Only update currentClipId if it matches the transcribed clip
  // This prevents changing currentClipId when user is viewing a different clip
  if (currentClipId === clipId) {
    // User is viewing this clip - safe to update
    setCurrentClipId(clipId);
  } else {
    // User is viewing a different clip - don't change currentClipId
    log.debug('Background transcription - not updating currentClipId', {
      clipId,
      currentClipId,
      reason: 'User viewing different clip'
    });
  }

  // Update clip content in storage
  updateClip(clipId, {
    content: transcription,
    status: 'transcribed'
  });

  // Start background formatting
  formatText(transcription, clipId);
}, [currentClipId, updateClip, formatText]);
```

**Why this works:**

```
Before fix:
1. User viewing Recording 01 (currentClipId = "clip-ABC")
2. Recording 03 transcribes (clipId = "clip-XYZ")
3. handleTranscriptionSuccess sets currentClipId = "clip-XYZ" ❌
4. UI re-renders, shows Recording 03's content in Recording 01's screen

After fix:
1. User viewing Recording 01 (currentClipId = "clip-ABC")
2. Recording 03 transcribes (clipId = "clip-XYZ")
3. handleTranscriptionSuccess checks: currentClipId !== clipId
4. Doesn't update currentClipId ✅
5. UI stays on Recording 01, doesn't show Recording 03's content
```
