## 🆕 PART 2: NEW ISSUE DISCOVERED

### 🔴 CRITICAL: Missing Module-Level State Issue

#### Issue 6.2: Move audioRetrievalAttempts to Zustand

**Status**: 🆕 **NEW CRITICAL ISSUE** (NOT in v6.10!)

**File**: [ClipMasterScreen.tsx:60](ClipMasterScreen.tsx#L60)

**Code**:
```typescript
// ❌ MODULE-LEVEL STATE (same anti-pattern as isProcessingPending)
const audioRetrievalAttempts = new Map<string, number>();
```

**Why This is a Problem**:
1. **Same issue as isProcessingPending** - module-level state in React component file
2. **Breaks React rendering model** - survives component unmount/remount
3. **HMR bugs** - state persists across hot reloads
4. **Testing issues** - state leaks between tests
5. **Memory leaks** - entries never cleared unless manually deleted

**Current Usage**:
- Tracks failed audio retrieval attempts per clip
- Used to prevent infinite retry loops
- Referenced at multiple points in ClipMasterScreen.tsx

**Fix** (Same pattern as Issue 6.1):
```typescript
// In clipStore.ts:
interface ClipStore {
  // ... existing fields
  audioRetrievalAttempts: Map<string, number>;
  setAudioRetrievalAttempt: (clipId: string, attempts: number) => void;
  clearAudioRetrievalAttempt: (clipId: string) => void;
}

// In ClipMasterScreen.tsx:
const audioRetrievalAttempts = useClipStore((state) => state.audioRetrievalAttempts);
const setAudioRetrievalAttempt = useClipStore((state) => state.setAudioRetrievalAttempt);
const clearAudioRetrievalAttempt = useClipStore((state) => state.clearAudioRetrievalAttempt);
```

**Priority**: 🔴 **CRITICAL** (Same as Issue 6.1)

**Why v6.10 Missed This**:
- Focused on `isProcessingPending` as the main module-level state
- `audioRetrievalAttempts` is defined nearby (line 60 vs 57) but was overlooked
- Both need the same fix
