## Error Breakdown

### Error #1: Line 85 - 'pending' Status Doesn't Exist

**Code**:
```typescript
} else if (currentClip.status === 'pending' || currentClip.status === 'pending-child') {
```

**Error**:
> This comparison appears to be unintentional because the types have no overlap.

**Why**:
- `currentClip` is typed as `clipStore.Clip`
- `clipStore.ClipStatus` does NOT include `'pending'`
- Only includes: `null | 'transcribing' | 'formatting' | 'pending-child' | 'pending-retry' | 'failed'`

**The Issue**:
- This check for `'pending'` is **legacy code** from when clipStorage.Clip was used
- In the new system:
  - **Parents** have `status: null` (they're just containers)
  - **Children** have `status: 'pending-child' | 'transcribing' | 'formatting' | 'failed' | null`
- There is NO `'pending'` status in the new type system

**The Fix**:
Remove the `'pending'` check entirely. It's dead code.

```typescript
} else if (currentClip.status === 'pending-child') {
  // This is a child clip - find its parent
  const actualParent = currentClip.parentId
    ? getClips().find(c => c.id === currentClip.parentId)
    : currentClip;
  shouldCreateParent = false;
  parentClipForChild = actualParent || currentClip;
```

### Error #2: Line 112 - Type Incompatibility

**Code**:
```typescript
const nextNumber = getNextRecordingNumber(getClips());
```

**Error**:
> Argument of type 'clipStore.Clip[]' is not assignable to parameter of type 'clipStorage.Clip[]'.
> Type 'formatting' is not assignable to type 'transcribing' | 'pending-child' | 'failed' | 'pending' | null.

**Why**:
- `getClips()` returns `clipStore.Clip[]` (NEW type)
- `getNextRecordingNumber()` expects `clipStorage.Clip[]` (OLD type)
- The status unions are incompatible:
  - clipStore has `'formatting'` and `'pending-retry'`
  - clipStorage has `'pending'`

**What getNextRecordingNumber Actually Does** (lines 274-288):
```typescript
export function getNextRecordingNumber(clips: Clip[]): string {
  const parentRecordings = clips.filter(c =>
    c.status !== 'pending-child' &&  // Uses status
    !c.parentId                       // Uses parentId
  );

  const recordingNumbers = parentRecordings
    .map(c => {
      const match = c.title.match(/^Recording (\d+)$/);  // Uses title
      return match ? parseInt(match[1], 10) : 0;
    })
```

**Analysis**:
The function only uses:
- `c.status` (to check if !== 'pending-child')
- `c.parentId` (to check if undefined)
- `c.title` (to extract number)

All these fields exist in both types! The issue is **purely type system**, not runtime.

**The Fix - Type Assertion**:
```typescript
const nextNumber = getNextRecordingNumber(getClips() as any);
```

**Why this is safe**:
- Both Clip types have the required fields (`status`, `parentId`, `title`)
- The function doesn't care about the other differences
- This is a temporary bridge while the codebase migrates from clipStorage to clipStore

## Why These Errors Exist

**Context**: The codebase is migrating from clipStorage (sessionStorage-based) to clipStore (Zustand-based).

- **clipStorage.Clip** = OLD interface (optional rawText/formattedText, has 'pending' status)
- **clipStore.Clip** = NEW interface (required rawText/formattedText, has 'formatting'/'pending-retry' statuses)

**useOfflineRecording.ts** is caught in the middle:
- Uses clipStore.Clip (new) for types
- Calls clipStorage.getNextRecordingNumber() (old) function
- Creates clips without required new fields

This is a **migration incomplete** issue, not a design flaw.
