## 🐛 BUG #1: Recording 01 Stuck in "Waiting to Transcribe"

### What Happened

**User's Scenario:**
1. Recorded while online
2. Pressed Done button during processing
3. Navigated back to home screen
4. Recording appeared as "Recording 01" with "Waiting to transcribe" status
5. **Never retried** - stuck forever

### Root Cause Analysis

**Two Separate Retry Systems Exist:**

#### System 1: Live Recording Retry (useClipRecording.ts)
**File:** [useClipRecording.ts:408-442](../../hooks/useClipRecording.ts#L408-L442)

**How it works:**
```typescript
// Network errors retry automatically WITHIN the same recording session
if (shouldRetry) {
  const nextRetryCount = retryCount + 1;

  if (nextRetryCount < MAX_RAPID_ATTEMPTS) {
    // Rapid retry: immediate
    retryTimerRef.current = setTimeout(() => transcribeRecording(), 0);
    return { text: '', error: 'network' };
  } else {
    // Interval retry: scheduled with setTimeout
    retryTimerRef.current = setTimeout(() => {
      setIsActiveRequest(true);
      setTranscriptionError(null);
      transcribeRecording();
    }, waitTime);
    return { text: '', error: 'network' };
  }
}
```

**Key behavior:**
- Retries happen WITHIN the same function call via setTimeout
- Does NOT create pending clips
- Only works while user stays on record screen
- **Stops retrying if user presses Done or navigates away**

#### System 2: Pending Clip Retry (processAllPendingClips)
**File:** [ClipMasterScreen.tsx:459-682](ClipMasterScreen.tsx#L459-L682)

**How it works:**
- For clips that were recorded offline or had network errors
- Creates `pending-child` clips
- **Only triggered by auto-retry** (online/offline events)

**When pending clips are created:**

**1. User is offline** (line 802):
```typescript
if (!isOnline) {
  handleOfflineRecording({
    audioId: recordedAudioId!,
    duration: recordedDuration,
    currentClipId
  });
  return;
}
```

**2. VPN blocking detected** (line 828):
```typescript
if (transcriptionError === 'dns-block') {
  console.log('[HandleDone] VPN blocking transcription, saving as pending');
  handleOfflineRecording({
    audioId: recordedAudioId!,
    duration: recordedDuration,
    currentClipId
  });
  return;
}
```

**3. Network errors detected** (line 844):
```typescript
if (transcriptionError === 'network') {
  console.log('[HandleDone] Transcription failed, saving as pending');
  handleOfflineRecording({
    audioId: recordedAudioId!,
    duration: recordedDuration,
    currentClipId
  });
  return;
}
```

### When processAllPendingClips is Triggered

**File:** [useAutoRetry.ts:20-46](../../hooks/useAutoRetry.ts#L20-L46)

```typescript
const handleOnline = async () => {
  isOnline = true;
  console.log('[Auto-Retry] Came online');

  const clips = useClipStore.getState().clips;
  const hasPendingClips = clips.some(c =>
    c.audioId && c.status === 'pending-child'

```

**ONLY TRIGGERS ON:**
1. `window 'online'` event (user goes from offline → online)
2. On mount if already online with pending clips (line 59-68)

### The Bug

**What happens when user:**
1. Records while online
2. VPN manipulation causes 3 rapid failures
3. useClipRecording.ts sets `transcriptionError = 'network'`
4. **User presses Done button before interval retry completes**
5. ClipMasterScreen.tsx handleDone detects `transcriptionError === 'network'` (line 842)
6. Calls handleOfflineRecording → creates pending-child clip
7. **User navigates back to home screen**

**Result:**
- Pending clip exists with status 'pending-child'
- **BUT auto-retry never fires** because:
  - User is already online (no offline→online transition)
  - No window 'online' event will fire
  - processAllPendingClips never gets called
- **Clip stuck in "waiting to transcribe" forever**

### Why This is Critical

**No Recovery Path:**
- User can't manually trigger retry
- Clip stays pending until user:
  1. Manually goes offline (disables WiFi)
  2. Goes back online
  3. THEN auto-retry fires

**This is terrible UX** - user has to know the workaround!

### Fix Required

**Option 1: Trigger retry on mount for all online pending clips** ✅ **RECOMMENDED**

Currently useAutoRetry.ts line 59-68 does this, but need to verify it's working:

```typescript
// On mount: If already online and have pending clips, start immediately

    c.audioId && c.status === 'pending-child'

    console.log('[Auto-Retry] Already online with pending clips, starting retry');
    processAllPendingClips();  // ← This should trigger on next app mount

```

**Testing hypothesis:**
- User presses Done → creates pending clip → navigates to home
- ClipMasterScreen unmounts → useAutoRetry unmounts
- Home screen doesn't mount useAutoRetry
- **useAutoRetry only runs in ClipMasterScreen context!**

**Let me verify where useAutoRetry is mounted...**

**File:** [ClipMasterScreen.tsx:1522-1525](ClipMasterScreen.tsx#L1522-L1525)


**CONFIRMED: useAutoRetry only runs in ClipMasterScreen!**

When user navigates to home screen:

- Event listeners removed
- **No retry mechanism exists on home screen**

When user navigates back to ClipMasterScreen:
- useAutoRetry mounts again
- Line 59-68 should trigger processAllPendingClips
- **This should work!**

**So why is it stuck?**

Possible reasons:
1. ClipMasterScreen hasn't been re-mounted since creating the pending clip
2. User stayed on home screen
3. Need to navigate back to record screen to trigger retry

**Fix:** Move useAutoRetry to **App.tsx** (always mounted) instead of ClipMasterScreen

## 🐛 BUG #2: Clipboard Copy During Auto-Retry

### What Happened

**User's Scenario:**
1. VPN manipulation caused 3 rapid failures
2. Became pending clip
3. Eventually succeeded (4th attempt)
4. **Got clipboard error:**
```
Runtime NotAllowedError
Failed to execute 'writeText' on 'Clipboard': Document is not focused.

ClipMasterScreen.tsx (1218:29)
```

### Root Cause

**File:** [ClipMasterScreen.tsx:1177-1221](ClipMasterScreen.tsx#L1177-L1221)

**formatTranscriptionInBackground callback:**
```typescript
const formatTranscriptionInBackground = useCallback((clip: Clip, isAppending: boolean) => {
  // ... formatting logic ...

  formatTextInBackground(textToFormat).then(formattedText => {
    // ... update clip ...

    // ❌ BUG: ALWAYS copies to clipboard, even for auto-retry!
    const textToCopy = isAppending
      ? updatedClip.rawText
      : updatedClip.formattedText;

    navigator.clipboard.writeText(textToCopy);  // ← LINE 1218

    setShowCopyToast(true);
  });
}, [/* ... */]);
```

**Why it fails:**
- Browser security: Clipboard API requires **document focus** or **recent user interaction**
- Auto-retry happens in background (no user interaction)
- User might be on different tab or home screen
- Browser blocks clipboard access

### When Clipboard Copy Should Happen

**Should copy:**
✅ User just finished recording on record screen (user interaction context)
✅ User is actively on record screen when transcription completes

**Should NOT copy:**
❌ Auto-retry processing pending clips in background
❌ User is on home screen or different tab
❌ No recent user interaction

### Fix Required

**Check if copy should happen before attempting:**

```typescript
// Determine if we should copy to clipboard
const shouldCopyToClipboard = (
  // User is on record screen (ClipMasterScreen mounted)
  recordNavState !== 'home' &&
  // Not an auto-retry scenario
  !clip.parentId  // If clip has parentId, it's a child from auto-retry
);

if (shouldCopyToClipboard) {
  try {
    const textToCopy = isAppending
      ? updatedClip.rawText
      : updatedClip.formattedText;

    navigator.clipboard.writeText(textToCopy);
    setShowCopyToast(true);
  } catch (error) {
    // Silently fail - clipboard access denied
    console.warn('[Clipboard] Failed to copy:', error);
  }
}
```

**Alternative: Only copy for live recordings, not pending clips:**

```typescript
// Only copy if this wasn't a pending clip retry
if (clip.status !== 'pending-child' && clip.status !== 'pending-retry') {
  try {
    navigator.clipboard.writeText(textToCopy);
    setShowCopyToast(true);
  } catch (error) {
    console.warn('[Clipboard] Failed to copy:', error);
  }
}
```

## 🔍 DNS-BLOCK ERROR: What Triggers It?

### User's Question

> "With regards to the VPN blocking bug, does it trigger when it finds an 'ECONNREFUSED' or DNS issue? What are the error messages that trigger it from the response via the API?"

### Answer: TypeError "Failed to fetch"

**File:** [transcriptionRetry.ts:193-198](../../utils/transcriptionRetry.ts#L193-L198)

```typescript
} catch (error) {
  // DNS-level blocking (VPN, corporate firewall, DNS hijacking)
  if (error instanceof TypeError && error.message.includes('Failed to fetch')) {
    logger.warn('[TranscriptionRetry]', 'DNS block detected', { attempt });
    return { text: '', error: 'dns-block' };
  }
  // ... other error handling
}
```

**What triggers it:**
- **Network-level blocking** that prevents request from reaching server
- DNS resolution failure
- VPN blocking the domain
- Corporate firewall blocking
- Connection refused at network level (before HTTP)

**Browser Fetch API behavior:**
- When network request **cannot be made**, throws `TypeError`
- Error message is **"Failed to fetch"** (generic browser message)
- This happens BEFORE any HTTP response

**NOT triggered by:**
❌ HTTP errors (500, 503, etc.) - these return `'api-down'` instead
❌ Timeout errors - these return `'network'` instead
❌ CORS errors - different error message

### Why You Didn't See dns-block in Your Test

**Your debug file showed:**
```
POST http://localhost:3000/api/clipperstream/transcribe 500 (Internal Server Error)
[TranscriptionRetry] [WARN] API down detected {attempt: 1}
```

**This is `'api-down'`, not `'dns-block'` because:**
1. Request **reached** the server (localhost:3000)
2. Server responded with 500 error
3. Code correctly categorized as 'api-down' (line 73-77 in transcriptionRetry.ts)

**Why VPN didn't block localhost:**
- Most VPNs route **external traffic**, not localhost
- localhost (127.0.0.1) typically bypasses VPN tunnel
- VPN has no reason to block loopback interface

### How to Actually Trigger dns-block

**Option 1: Use external API endpoint**
- Deploy API to external domain (e.g., Vercel, Netlify)
- Enable VPN that blocks that domain
- Record → Should trigger dns-block

**Option 2: Browser DevTools offline mode**
- Open DevTools → Network tab
- Enable "Offline" mode BEFORE recording
- Attempt transcription → Should trigger "Failed to fetch"

**Option 3: Modify hosts file + VPN** (advanced)
```bash
# Mac/Linux: /etc/hosts
# Windows: C:\Windows\System32\drivers\etc\hosts
127.0.0.1 blocked.localhost
```
Then change API endpoint to `blocked.localhost:3000` and use VPN

**Option 4: Network firewall rule**
- Block localhost:3000 in firewall settings
- Attempt transcription → Should trigger dns-block

### ECONNREFUSED vs Failed to fetch

**ECONNREFUSED:**
- Node.js error (server-side)
- Means server tried to connect to something that refused
- You won't see this in browser

**Failed to fetch:**
- Browser error (client-side)
- Generic network failure message
- This is what triggers dns-block

## 🤔 AUTO-RETRY vs RETRY LOGIC: User's Confusion

### User's Question

> "To my understanding, auto-retry is meant to let you know if you're online or not and then kick the retry logic to go into interaction. But if it's a case whereby you never went offline and the recording just failed because you were playing with your VPN to force it to fail and it tried three attempts and then it moves it to pending clips and then obviously it got translated after being a pending clip for a while right? The question is, does that fall under the category of auto-retry or under the category of just the plain retry logic itself?"

### Answer: It's COMPLICATED (Two Systems)

#### Scenario Breakdown

**What actually happened in your test:**

**Phase 1: Live Recording Retry (NOT auto-retry)**
1. Started recording while online
2. VPN manipulation caused failures
3. **useClipRecording.ts built-in retry** kicked in (lines 408-442)
4. Attempted 3 rapid retries
5. **Would have scheduled interval retry** BUT...

**Phase 2: User Pressed Done Early**
6. User pressed Done button before interval retry could complete
7. handleDone detected `transcriptionError === 'network'` (line 842)
8. Called handleOfflineRecording → **created pending-child clip**
9. User navigated to home screen

**Phase 3: Stuck (No Auto-Retry Trigger)**
10. Clip is now pending with status 'pending-child'
11. **BUT user never went offline/online** → no auto-retry trigger
12. **useAutoRetry only runs in ClipMasterScreen** → unmounted when on home screen
13. **Clip stuck until user returns to ClipMasterScreen**

### The Two Systems Explained

#### System 1: Built-In Retry (useClipRecording.ts)
**Purpose:** Retry transcription for live recordings
**Trigger:** Automatic after network error
**Mechanism:** setTimeout scheduling
**Runs while:** User stays on record screen
**Stops when:** User navigates away or presses Done

**Does NOT involve:**
- Auto-retry hook
- processAllPendingClips
- Online/offline events

#### System 2: Auto-Retry (useAutoRetry.ts + processAllPendingClips)
**Purpose:** Process pending clips when user comes online
**Trigger:** Online/offline events OR mount with pending clips
**Mechanism:** processAllPendingClips in while loop
**Runs while:** ClipMasterScreen is mounted
**Processes:** pending-child clips created by handleOfflineRecording

**Requires:**
- Pending clips exist (status === 'pending-child')
- Online event fires OR ClipMasterScreen mounts

### Direct Answer

**Your scenario involved BOTH systems:**

1. **Built-in retry** (useClipRecording.ts) ran first:
   - Attempted 3 rapid retries
   - Would have done interval retry BUT...

2. **User interrupted by pressing Done:**
   - Created pending clip
   - System 1 stopped (user left record screen)

3. **Auto-retry DID NOT RUN** because:
   - Never went offline/online (no event trigger)
   - ClipMasterScreen unmounted (useAutoRetry removed)
   - Clip stuck pending

**If user returns to record screen:**
- ClipMasterScreen mounts
- useAutoRetry mounts
- Line 59-68: Checks for pending clips
- **Should trigger processAllPendingClips**
- This is auto-retry (System 2) finishing what System 1 started

### Why It's Confusing

**The two systems create a gap:**
- System 1 handles live recording retries
- System 2 handles offline recording retries
- **Gap:** When live recording fails → creates pending → needs System 2 but System 2 only runs in ClipMasterScreen

**The handoff is broken:**
- System 1 creates pending clip and says "System 2 will handle it"
- System 2 never runs because user left the screen
- Clip stuck in limbo
