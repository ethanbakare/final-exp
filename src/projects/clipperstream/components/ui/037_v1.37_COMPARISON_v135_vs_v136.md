## Detailed Analysis

### 1. Architecture Pattern

#### v1.35: Circuit Breaker (CLOSED/OPEN/HALF_OPEN)

**How it works**:
```typescript
class CircuitBreaker {
  state: 'CLOSED' | 'OPEN' | 'HALF_OPEN'

  shouldTryDeepgram(isQueueRetry) {
    if (isQueueRetry && state === 'OPEN') return false;
    if (!isQueueRetry && state === 'OPEN') return true;
    // ... complex state transitions
  }
}
```

**Strengths**:
- ✅ Industry-standard pattern (used by Netflix, AWS, etc.)
- ✅ Formal state machine (predictable behavior)
- ✅ Distinguishes queue retries from new recordings
- ✅ New recordings always retry Deepgram (optimistic)
- ✅ Self-healing via HALF_OPEN state

**Weaknesses**:
- ❌ Overkill for this use case
- ❌ More complex to understand
- ❌ Harder to debug (3-state system)
- ❌ More code to maintain
- ❌ New recordings retry Deepgram even if VPN is still on (wasted time)

---

#### v1.36: Session-Based Fallback (Boolean Flag)

**How it works**:
```typescript
let useWhisperFallback = false;
let lastFallbackTime = 0;

if (Date.now() - lastFallbackTime > 5min) {
  useWhisperFallback = false;  // Reset
}

if (useWhisperFallback) {
  return await whisper(...);
} else {
  try { return await deepgram(...); }
  catch (dns_error) { useWhisperFallback = true; return await whisper(...); }
}
```

**Strengths**:
- ✅ Simple, easy to understand
- ✅ Minimal code footprint
- ✅ Single file modification
- ✅ Time-based reset (user-friendly: 5 min = likely session change)
- ✅ Treats all requests equally (consistent behavior)
- ✅ More comprehensive error detection

**Weaknesses**:
- ❌ Not industry-standard pattern (custom approach)
- ❌ New recordings don't retry Deepgram (might miss VPN-off state)
- ❌ No explicit queue-empty reset (relies on time only)

### 3. New Recording Behavior During Queue Processing

**Scenario**: User records NEW audio while queue is processing 5 pending clips (VPN ON)

#### v1.35 Behavior:

```
Queue Processing (background):
├─ Clip 1 (retry): Try Deepgram → Fail → Circuit OPENS → Whisper (3s)
├─ Clip 2 (retry): Skip Deepgram (circuit OPEN) → Whisper (2s)
└─ Clip 3 (retry): Skip Deepgram → Whisper (2s)

User Records NEW (foreground):
└─ Clip 4 (new): Try Deepgram (circuit allows for new recordings) → Fail → Whisper (3s)
                 ↑ WASTED TIME (1s Deepgram timeout)
```

**Result**: New recording wastes time retrying Deepgram

---

#### v1.36 Behavior:

```
Queue Processing (background):
├─ Clip 1 (retry): Try Deepgram → Fail → Fallback ON → Whisper (3s)
├─ Clip 2 (retry): Fallback ON → Whisper (2s)
└─ Clip 3 (retry): Fallback ON → Whisper (2s)

User Records NEW (foreground):
└─ Clip 4 (new): Fallback ON → Whisper directly (2s)
                 ↑ NO WASTED TIME
```

**Result**: New recording uses Whisper directly (faster)

**Winner**: **v1.36** - Faster UX, doesn't waste time

### 4. Reset Strategy

#### v1.35: Manual + Timeout

**How it resets**:
1. When queue empties: `circuitBreaker.reset()` (manual)
2. After 60 seconds: Transitions to HALF_OPEN (auto)

**Pros**:
- ✅ Queue-empty reset is smart (VPN might be off by then)
- ✅ 60s timeout is quick recovery

**Cons**:
- ❌ Requires client to call reset (tight coupling)
- ❌ What if client forgets to reset? (state gets stuck)

---

#### v1.36: Time-Based Only

**How it resets**:
1. After 5 minutes of inactivity: Auto-reset to Deepgram

**Pros**:
- ✅ Fully automatic (no client involvement)
- ✅ 5 minutes = reasonable session boundary
- ✅ User likely changed VPN state after 5min idle

**Cons**:
- ❌ Doesn't reset immediately when queue empties
- ❌ If VPN is still on after 5min, wastes one more Deepgram try

**Winner**: **v1.36** - More autonomous (doesn't rely on client)

## Critical Difference: New Recording Logic

**This is the MOST important architectural difference.**

### Scenario: Queue Processing, User Records New Audio

**v1.35 Logic**:
```typescript
// New recording always tries Deepgram first
shouldTryDeepgram(isQueueRetry=false) → returns true (even if circuit OPEN)
```

**Reasoning**: "Fresh user action might have better network state"

**Problem**: If VPN is still on (which it usually is), wastes 1-3 seconds on Deepgram timeout

---

**v1.36 Logic**:
```typescript
// New recording checks fallback state
if (useWhisperFallback && timeSinceLastFallback < 5min) {
  return await transcribeWithWhisper(...);  // Skip Deepgram
}
```

**Reasoning**: "If VPN is still active, Deepgram will fail again. Save time."

**Benefit**: Instant feedback (no wasted Deepgram timeout)

---

**Which is better?**

**Answer**: **v1.36 is more pragmatic**

**Why**:
1. **VPN state doesn't change mid-session**: Users typically don't toggle VPN while using the app
2. **User frustration**: Waiting 3s for Deepgram to fail EVERY time is bad UX
3. **Optimistic retry is risky**: 99% chance VPN is still on → 99% wasted time

**v1.35's optimistic approach is theoretically sound**, but **practically wasteful**.
