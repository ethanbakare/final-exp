## THE PROBLEM

### Race Condition with Global `isActiveRequest`

The spinner timing issue occurs because `isActiveRequest` from `useClipRecording` is a **SINGLE GLOBAL FLAG**, but we're processing **MULTIPLE CLIPS IN PARALLEL**:

- HTTP is **sequential** (await in loop)
- Formatting is **parallel** (async background processing)

**Timeline of the Bug**:
```
Time 0ms:    Clip 1 HTTP starts → isActiveRequest = true
Time 500ms:  Clip 1 HTTP completes → Loop moves to Clip 2
             Clip 1 formatting starts in background (async)
Time 501ms:  Clip 2 HTTP starts → isActiveRequest = true (still)
Time 1000ms: Clip 1 formatting completes → resetRecording() → isActiveRequest = FALSE
             ⚠️ BUT CLIP 2 IS STILL DOING HTTP!
             ⚠️ Parent A spinner stops incorrectly
Time 1500ms: Clip 2 HTTP completes → Loop moves to Clip 3
             Clip 2 formatting starts in background
Time 2000ms: Clip 2 formatting completes → resetRecording() → isActiveRequest = FALSE
             ⚠️ BUT CLIP 3 IS STILL DOING HTTP!
```

**Result**: First clip spins briefly, then all clips show static "Transcribing..." even though HTTP is ongoing.

## THE SOLUTION

### Per-Clip HTTP Tracking with `activeHttpClipId`

Track which **SPECIFIC CLIP** is currently doing HTTP, independent of the global `isActiveRequest` flag.

**Key Points**:
1. Set `activeHttpClipId = clip.id` IMMEDIATELY before `transcribeRecording()`
2. Clear `activeHttpClipId = null` IMMEDIATELY after `transcribeRecording()` completes
3. This state is IMMUNE to `resetRecording()` calls from other clips
4. Parent's spinner = "Do I have a child where `child.id === activeHttpClipId`?"

**Timeline with Fix**:
```
Time 0ms:    activeHttpClipId = "clip-1"
             Clip 1 HTTP starts → Parent A spinner ON
Time 500ms:  Clip 1 HTTP completes → activeHttpClipId = null
             Clip 1 formatting starts (async)
             activeHttpClipId = "clip-2"
             Clip 2 HTTP starts → Parent A spinner STILL ON (same parent, different child)
Time 1000ms: Clip 1 formatting completes → resetRecording() (global flag reset)
             ⚠️ BUT activeHttpClipId = "clip-2" (unaffected!)
             ✅ Parent A spinner STAYS ON
Time 1500ms: Clip 2 HTTP completes → activeHttpClipId = null
             Clip 2 formatting starts (async)
             activeHttpClipId = "clip-3"
             Clip 3 HTTP starts → Parent A spinner STILL ON
```

## ARCHITECTURAL NOTES

### Why This Works

1. **HTTP tracking is independent of formatting**: `setActiveHttpClipId()` happens in the loop, not in async callbacks
2. **Immune to resetRecording()**: `activeHttpClipId` is a separate state variable, unaffected by `resetRecording()`
3. **Per-clip granularity**: Each clip gets its own tracking, not shared global state
4. **Parent-level derivation**: ClipHomeScreen derives parent state from children's HTTP activity

### State Hierarchy

```
ClipMasterScreen (owns state):
  ├─ activeTranscriptionParentId: Which parent should show ANY activity?
  └─ activeHttpClipId: Which specific child is doing HTTP RIGHT NOW?
       ↓
  ClipHomeScreen (derives display state):
  └─ derivedIsActiveRequest = children.some(c => c.id === activeHttpClipId)
```

### Why Not Use Global `isActiveRequest`?

**Problem**: `isActiveRequest` from `useClipRecording` is reset by ANY call to `resetRecording()`, which happens when:
- Active recording completes
- Background transcription formatting completes
- Manual retry completes

**Solution**: Track HTTP activity separately from recording session state.
