## EXECUTIVE SUMMARY

### The Complete Picture

**useClipRecording.ts** (The Engine):
- Does HTTP requests for transcription
- Manages retry logic (rapid phase + interval phase)
- Controls `isActiveRequest` flag (true during HTTP, false during wait intervals)
- Used by BOTH active recordings AND auto-retry queue

**ClipMasterScreen.tsx** (The Orchestrator):
- Runs auto-retry loop for queued clips
- Calls `transcribeRecording(blob)` which triggers useClipRecording
- **PROBLEM**: Has `activeHttpClipId` state but NEVER sets it
- **SOLUTION**: Set/clear `activeHttpClipId` around HTTP calls

**ClipHomeScreen.tsx** (The Display):
- Derives parent spinner from children
- **ALREADY FIXED** in v2.5.4: Uses `activeHttpClipId` to match children
- Waiting for ClipMasterScreen to actually SET the value

**useTranscriptionHandler.ts** (The Success Handler):
- Handles transcription completion
- **TWO BUGS**:
  1. Race condition: checks `c.content` which isn't set until AFTER formatting
  2. Off-by-one: flushes batch at `remaining === 1` instead of `=== 0`

## THE THREE BUGS

### Bug #1: Missing activeHttpClipId Set/Clear (CRITICAL)
**Location**: ClipMasterScreen.tsx auto-retry loop (around line 537)

**Impact**: Spinner never rotates during auto-retry because ClipHomeScreen's derivation can't find matching child

**Evidence**:
- Line 146: `const [activeHttpClipId, setActiveHttpClipId] = useState<string | null>(null);` ✅ State exists
- Auto-retry loop: **NEVER calls** `setActiveHttpClipId(clip.id)` ❌
- ClipHomeScreen line 173: `children.some(c => c.id === activeHttpClipId)` → Always false because activeHttpClipId is always null

### Bug #2: Race Condition in isFirstPendingForClip
**Location**: useTranscriptionHandler.ts:104

**Current Code**:
```typescript
const hasContent = allForParent.some(c => c.content);
```

**Problem**:
- Checks `c.content` field
- `c.content` is only set AFTER async `formatTranscriptionInBackground()` completes
- When Child 2 HTTP completes, Child 1 is still formatting → `hasContent = false`
- Result: Child 2 incorrectly treated as "first" → formats immediately instead of batching

**Timeline**:
```
T0: Child 1 HTTP completes → starts formatting (async)
T1: Child 2 HTTP completes → checks hasContent
    → Child 1.content still undefined (formatting not done)
    → hasContent = false
    → Child 2 treated as "first" ❌
    → Both format in parallel instead of batching
```

### Bug #3: Off-by-One in Batch Flush Condition
**Location**: useTranscriptionHandler.ts:254

**Current Code**:
```typescript
if (remaining === 1) {  // This is the last one
```

**Problem**:
- `countRemainingPending()` EXCLUDES current clip (line 125: `c.id !== clip.id`)
- Last clip (Clip 4) has 0 remaining siblings, not 1
- Condition `remaining === 1` means "flush when second-to-last completes"
- Result: Batch never flushes, clips 2-4 never format

**Evidence** (4-clip scenario):
```
Clip 1: remaining = 3 (clips 2,3,4) → immediate format ✅
Clip 2: remaining = 2 (clips 3,4) → batch (wait for remaining === 1)
Clip 3: remaining = 1 (clip 4) → batch (SHOULD flush here but doesn't)
Clip 4: remaining = 0 → batch (NEVER flushes because remaining !== 1)
```

**Correct Logic**:
```typescript
if (remaining === 0) {  // This is the last one
```

## THE FIXES

### Fix #1: Set/Clear activeHttpClipId in Auto-Retry Loop

**File**: ClipMasterScreen.tsx
**Location**: Around line 537 (inside auto-retry loop)

**Find This** (approx lines 537-560):
```typescript
for (const clip of pendingClips) {
  log.debug('Processing pending clip from queue', {
    clipId: clip.id,
    hasAudio: !!clip.audioId
  });

  if (clip.audioId) {
    try {
      const audioBlob = await getAudio(clip.audioId);
      if (audioBlob) {
        log.info('Retrying transcription with saved audio', {
          clipId: clip.id,
          audioId: clip.audioId
        });

        await transcribeRecording(audioBlob);
      } else {
        log.warn('Audio blob not found', { audioId: clip.audioId });
      }
    } catch (error) {
      log.error('Failed to retrieve audio', { error, clipId: clip.id });
    }
  }
}
```

**Replace With**:
```typescript
for (const clip of pendingClips) {
  log.debug('Processing pending clip from queue', {
    clipId: clip.id,
    hasAudio: !!clip.audioId
  });

  if (clip.audioId) {
    try {
      const audioBlob = await getAudio(clip.audioId);
      if (audioBlob) {
        log.info('Retrying transcription with saved audio', {
          clipId: clip.id,
          audioId: clip.audioId
        });

        // v2.5.6 FIX: Track which clip is doing HTTP RIGHT NOW
        setActiveHttpClipId(clip.id);

        try {
          await transcribeRecording(audioBlob);
        } finally {
          // v2.5.6 FIX: Clear HTTP tracking when done (success or failure)
          setActiveHttpClipId(null);
        }
      } else {
        log.warn('Audio blob not found', { audioId: clip.audioId });
      }
    } catch (error) {
      log.error('Failed to retrieve audio', { error, clipId: clip.id });
    }
  }
}
```

**Why This Works**:
- Sets `activeHttpClipId` BEFORE HTTP starts → ClipHomeScreen can match child
- Clears in `finally` block → Works even if HTTP fails
- Each clip gets its turn with the flag → Sequential HTTP tracking
- ClipHomeScreen derivation (line 173): `children.some(c => c.id === activeHttpClipId)` → Now returns true during this clip's HTTP

### Fix #2: Change Batch Flush Condition (Off-by-One)

**File**: useTranscriptionHandler.ts
**Location**: Line 254

**Find This**:
```typescript
        // If last one, display all batched
        if (remaining === 1) {  // This is the last one
```

**Replace With**:
```typescript
        // If last one, display all batched
        if (remaining === 0) {  // v2.5.6 FIX: Last clip has 0 remaining (countRemainingPending excludes current)
```

**Why This Works**:
- `countRemainingPending()` excludes current clip (line 125)
- Last clip has 0 siblings remaining
- Now correctly flushes batch when last clip completes

### Fix #3: Fix Race Condition in isFirstPendingForClip

**File**: useTranscriptionHandler.ts
**Location**: Line 104

**Option A: Check Status Instead of Content** (Recommended - Minimal Change)

**Find This**:
```typescript
    // First if no child has content yet
    const hasContent = allForParent.some(c => c.content);
```

**Replace With**:
```typescript
    // v2.5.6 FIX: Check if any child has COMPLETED (not just "has content")
    // A child is "completed" when status is cleared (null) AND has content
    // This prevents race where Child 2 checks while Child 1 is still formatting
    const hasCompleted = allForParent.some(c => c.status === null && c.content);
```

**Why This Works**:
- `status = null` is only set AFTER formatting completes
- Prevents Child 2 from checking while Child 1 is still formatting
- Child 1: HTTP done → `status = 'transcribing'` → formatting starts
- Child 2: HTTP done → checks `hasCompleted` → Child 1 still has `status = 'transcribing'` → false
- Result: Child 2 correctly batches

**Option B: Add Explicit Formatting Status** (More Robust - Requires More Changes)

Would require:
1. Add `status = 'formatting'` when formatting starts
2. Clear to `null` when formatting completes
3. Update status checks throughout codebase

**Recommendation**: Use Option A for surgical fix, consider Option B for future refactor

## WHY THESE FIXES WORK TOGETHER

### The Complete Flow (After Fixes):

**4-Clip Queue Example**:

```
CLIP 1:
  ├─ ClipMasterScreen: setActiveHttpClipId(clip1.id)
  ├─ useClipRecording: HTTP request
  ├─ ClipHomeScreen: children.some(c => c.id === clip1.id) ✅ → SPINNER ROTATES
  ├─ HTTP completes → transcription arrives
  ├─ useTranscriptionHandler: isFirstPending = true (hasCompleted = false)
  ├─ Formats immediately
  ├─ Sets status = null after formatting
  └─ ClipMasterScreen: setActiveHttpClipId(null) in finally

CLIP 2:
  ├─ ClipMasterScreen: setActiveHttpClipId(clip2.id)
  ├─ useClipRecording: HTTP request
  ├─ ClipHomeScreen: children.some(c => c.id === clip2.id) ✅ → SPINNER ROTATES
  ├─ HTTP completes → transcription arrives
  ├─ useTranscriptionHandler: isFirstPending = false (Clip 1 has status=null + content)
  ├─ remaining = 2 (clips 3,4)
  ├─ Adds to batch, doesn't flush (remaining !== 0)
  └─ ClipMasterScreen: setActiveHttpClipId(null) in finally

CLIP 3:
  ├─ ClipMasterScreen: setActiveHttpClipId(clip3.id)
  ├─ useClipRecording: HTTP request
  ├─ ClipHomeScreen: children.some(c => c.id === clip3.id) ✅ → SPINNER ROTATES
  ├─ HTTP completes → transcription arrives
  ├─ useTranscriptionHandler: isFirstPending = false
  ├─ remaining = 1 (clip 4)
  ├─ Adds to batch, doesn't flush (remaining !== 0)
  └─ ClipMasterScreen: setActiveHttpClipId(null) in finally

CLIP 4:
  ├─ ClipMasterScreen: setActiveHttpClipId(clip4.id)
  ├─ useClipRecording: HTTP request
  ├─ ClipHomeScreen: children.some(c => c.id === clip4.id) ✅ → SPINNER ROTATES
  ├─ HTTP completes → transcription arrives
  ├─ useTranscriptionHandler: isFirstPending = false
  ├─ remaining = 0 (no more siblings)
  ├─ v2.5.6 FIX: remaining === 0 → FLUSHES BATCH ✅
  ├─ Formats clips 2, 3, 4 sequentially
  ├─ Generates title after first child
  └─ ClipMasterScreen: setActiveHttpClipId(null) in finally
```

**Result**:
- ✅ All spinners rotate continuously during their HTTP phase
- ✅ First child formats immediately
- ✅ Remaining children batch and format together
- ✅ Title generates after first child completes
- ✅ No content bleeding between clips

**Impact**: User sees title sooner (after 1st child instead of all children)
