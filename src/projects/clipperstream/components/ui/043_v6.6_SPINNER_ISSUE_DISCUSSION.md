## тЭУ The Architectural Question

### Original Design Intent (Pre-v6)

**Before introducing `'retry-pending'` status**, there was only:
- `'transcribing'` = could mean "actively making HTTP request" OR "waiting between retries"

So `isActiveRequest` was needed to distinguish:
- `status='transcribing'` + `isActiveRequest=true` тЖТ "Actively transcribing" (spinner spins)
- `status='transcribing'` + `isActiveRequest=false` тЖТ "Waiting between retries" (spinner static)

### Current Design (Post-v6)

**Now we have distinct statuses**:
- `'transcribing'` = actively making HTTP request
- `'retry-pending'` = waiting between retries (moved to next parent)

**User's Point**: 
> "If `'transcribing'` ALWAYS means HTTP is happening, and `'retry-pending'` means we're waiting, then why do we need `isActiveRequest`? The status alone should be enough!"

## ЁЯФН Can We Track Active HTTP Inside Intervals?

### My Proposed Change
```typescript
setActiveHttpClipId(firstClip.id);     // Set at START
await attemptTranscription(audioBlob, {
  maxRapidAttempts: 3,
  useIntervals: true,
});
setActiveHttpClipId(null);             // Clear at END
```

### The Problem
**This doesn't work** because:
- `attemptTranscription` is ONE function call
- From the outside, you can't see inside it
- `activeHttpClipId` would be set for the ENTIRE 3-4 minutes
- Spinner would spin CONTINUOUSLY, even during 30s, 1min, 2min waits
- This is WRONG behavior

### Can We Track Individual HTTP Requests?

**NO, we cannot** - with the current architecture.

`attemptTranscription` doesn't have callbacks like:
```typescript
await attemptTranscription(audioBlob, {
  onHttpStart: () => setActiveHttpClipId(id),  // тЖР Doesn't exist
  onHttpEnd: () => setActiveHttpClipId(null),  // тЖР Doesn't exist
});
```

**Conclusion**: We cannot accurately track whether HTTP is actively happening inside the retry loop.

## ЁЯТб User's Proposed Solution

### The Argument

**User's Position**:
> "`isActiveRequest` is redundant. We should just have:
> - `'transcribing'` тЖТ spinner ALWAYS spins
> - `'retry-pending'` тЖТ spinner NEVER spins
>
> If the status is `'transcribing'`, it means we're actively doing something, so spin the spinner for the entire duration (including waits). That's simpler and doesn't require tracking `activeHttpClipId`."

### The Trade-off

**Option A: Keep `isActiveRequest` (Current)**
- Pro: Can distinguish active HTTP from waiting (in theory)
- Con: Doesn't actually work with current architecture
- Con: More complex state management
- Con: Redundant with new status system

**Option B: Remove `isActiveRequest` (User's Proposal)**
- Pro: Simpler - status alone controls spinner
- Pro: Matches the mental model of distinct statuses
- Pro: No need to track `activeHttpClipId`
- Con: Spinner spins during interval waits (30s, 1min, 2min)
- Con: User can't tell if HTTP is actively happening vs waiting
