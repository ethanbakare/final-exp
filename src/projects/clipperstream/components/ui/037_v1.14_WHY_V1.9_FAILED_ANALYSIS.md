## The Root Cause (Final Answer)

### The Complete Flow

1. **Recording offline**: Blob created with `type: 'audio/webm;codecs=opus'` ✅
2. **Store in IndexedDB**: `audioStorage.storeAudio()` saves blob + mimeType separately ✅
3. **Retrieve from IndexedDB**: `audioStorage.getAudio()` recreates blob with correct type ✅
4. **Pass to processChild**: `const audioBlob = await getAudio(...)` receives corrected blob ✅
5. **Pass to transcribeRecording**: `transcribeRecording(audioBlob)` receives blob... ⚠️
6. **Create FormData**: 
   ```typescript
   const formData = new FormData();
   formData.append('audio', blobToUse, 'recording-timestamp.webm');
   ```
   **If `blobToUse.type` is empty, FormData doesn't set proper Content-Type!** ❌

7. **Server receives**: File is there, but MIME type is wrong/missing ❌
8. **Deepgram rejects**: "This isn't valid audio" ❌
9. **500 error returned** ❌

### Why Blob.type Gets Lost

Between steps 4 and 5, the blob reference might:
- Be passed through an async boundary
- Get wrapped/unwrapped by React state
- Lose its type property due to browser internals

**JavaScript Blob quirk**: The `type` property is NOT guaranteed to be preserved when:
- Passing through async operations
- Storing in React state
- Cloning or copying the blob

## The Solution: Defensive Blob Handling

### Two-Layer Protection

**Layer 1**: Storage-level fix (audioStorage.ts) ✅ DONE
- Recreate blob when retrieving from IndexedDB
- Ensures `getAudio()` returns blob with correct type

**Layer 2**: Point-of-use fix (ClipMasterScreen.tsx) ❌ NOT DONE
- Verify blob.type before sending to API
- Recreate if needed using stored `child.mimeType`
- Add explicit logging

### Why BOTH Layers Are Needed

**Storage layer alone**: Trusts that blob.type will be preserved
**Point-of-use layer**: Doesn't trust anything, verifies at critical moment

**Industry practice**: Always verify critical properties before HTTP requests.
