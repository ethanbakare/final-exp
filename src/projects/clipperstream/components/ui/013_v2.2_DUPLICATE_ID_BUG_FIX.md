# 013_v2.2: Fix Duplicate Clip ID Bug

**Date:** December 27, 2025
**Version:** 2.2 (Critical bug fix for Phase 2.1)
**Issue:** Multiple pending clips share same ID causing React errors and display issues
**Time:** 5-10 minutes
**Complexity:** LOW (1 code block change)

---

## 🐛 Bug Discovery

**Reported symptoms:**
1. ✅ First offline recording → "Clip 001" appears
2. ✅ Second offline recording → "Clip 002" appears
3. ✅ Third offline recording → "Clip 003" appears
4. ❌ Navigate away (go to home screen)
5. ❌ Navigate back → Only "Clip 003" shows (001 and 002 disappeared!)
6. ❌ Record button works but X and Done buttons disappear

**Console errors (from 013_console_info.md line 56+):**
```
Encountered two children with the same key, `clip-1766863526800-bmcwndp0c`.
Keys should be unique so that components maintain their identity across updates.
```

This error repeats 10+ times!

---

## 🔍 Root Cause Analysis

**What's happening:**

All three pending clips (001, 002, 003) are using **THE SAME CLIP ID**.

```typescript
// Storage after 3 offline recordings:
clips = [
  {
    id: "clip-1766863526800-bmcwndp0c",  // ← SAME ID!
    pendingClipTitle: "Clip 003"          // ← Last update wins
  }
]

// State after 3 offline recordings:
selectedPendingClips = [
  { id: "clip-1766863526800-bmcwndp0c", title: "Clip 001" },  // ← SAME ID
  { id: "clip-1766863526800-bmcwndp0c", title: "Clip 002" },  // ← SAME ID
  { id: "clip-1766863526800-bmcwndp0c", title: "Clip 003" }   // ← SAME ID
]
```

**Why React complains:**
- React requires unique `key` prop for array items
- All three pending clips have `id: "clip-1766863526800-bmcwndp0c"`
- React's behavior with duplicate keys is "unsupported" → de-duplication, omission, undefined behavior
- Result: Only last clip renders, buttons disappear

**Why navigation restores only one clip:**

When you click the clip to navigate back:

```typescript
// handleClipClick (line 247-253):
const clip = clips.find(c => c.id === clipId);  // ← Finds the ONE clip in storage
setSelectedPendingClips([{
  id: clip.id,  // ← Same ID: clip-1766863526800-bmcwndp0c
  title: clip.pendingClipTitle  // ← "Clip 003" (last update)
}]);
```

Only ONE clip exists in storage (with the last `pendingClipTitle`), so only ONE clip is restored.

## 🎓 Lessons Learned

### Why the Bug Wasn't Obvious:

1. **The visual worked initially** - You could create 001, 002, 003
2. **State vs Storage mismatch** - State had 3 items, storage had 1
3. **React silently handled it** - Duplicate keys don't crash, just cause weird behavior
4. **Navigation revealed it** - Only then did storage become source of truth

### How Console Errors Helped:

1. **"Duplicate key" error** - Told us all clips shared same ID
2. **The ID in the error** - `clip-1766863526800-bmcwndp0c` → confirmed single ID
3. **Repetition (10+ times)** - Showed extent of problem

### Architectural Insight:

**Phase 2.1 assumed:**
- Multiple pending clips → `selectedPendingClips` array (correct)
- Each clip unique → shared same parent (incorrect assumption)

**Reality:**
- Each offline recording IS a separate clip
- They should have unique IDs
- `pendingClipTitle` is just a display name, not a differentiator

**The fix aligns code with reality.**
