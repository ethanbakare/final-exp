## What Each Output Will Prove

### If Client-Side Shows INVALID WebM
```
[AudioStorage] [ERROR] ❌ INVALID WEBM DATA
```
**Proves**: The blob is corrupted BEFORE sending to server
**Root cause**: Either:
- MediaRecorder created invalid audio while offline
- IndexedDB corrupted the data during storage
- ArrayBuffer → Blob conversion is wrong

**Next step**: Add logging to MediaRecorder to verify audio at recording time

---

### If Client-Side Shows VALID, Server-Side Shows INVALID
```
[AudioStorage] [INFO] ✅ VALID WEBM DATA
=== SERVER AUDIO VERIFICATION ===
Is valid WebM: ❌ NO
```
**Proves**: Blob is valid in browser but corrupted during upload
**Root cause**: Either:
- FormData upload corrupts the blob
- Server-side file write corrupts the data
- Blob → File conversion issue

**Next step**: Add logging to upload process

---

### If Both Show VALID, Deepgram Shows Specific Error
```
[AudioStorage] [INFO] ✅ VALID WEBM DATA
=== SERVER AUDIO VERIFICATION ===
Is valid WebM: ✅ YES
=== DEEPGRAM API ERROR DETAILS ===
Error code: ERR_INVALID_CODEC
```
**Proves**: Audio format is valid WebM but Deepgram doesn't support the codec
**Root cause**: MediaRecorder might use different codec when offline

**Next step**: Check MediaRecorder codec settings

---

### If Everything Shows VALID but Still Fails
```
✅ VALID WEBM DATA (client)
✅ VALID WEBM DATA (server)
=== DEEPGRAM API ERROR DETAILS ===
Error: fetch failed
```
**Proves**: WebM format is correct, but there's another issue
**Possible causes**:
- Deepgram API key issue
- Network connectivity to Deepgram
- WebM container valid but audio stream corrupted
- Deepgram rate limiting

**Next step**: Test with a known-good audio file
